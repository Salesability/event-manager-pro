## Codex Adversarial Review (re-pass)

### Verdict on prior High — **closed**

`updatePerson` off→on now creates the auth user, then conditionally claims the contact with `WHERE contacts.user_id IS NULL RETURNING id` at `src/features/people/actions.ts:492`. On `claimed.length === 0`:

1. Compensation branch at `:498` bans the just-created auth user at `:503` (`ban_duration='876000h'`, `app_metadata.role=null`).
2. Returns the race error at `:507` *before* `syncAuthMetadata` runs at `:513`.

So the prior bug — admin-stamping before compensation — is structurally impossible. B's auth user is *never* stamped admin in the race-loser path.

### Verdict on prior Medium — **closed**

`updatePerson` parses roles, then coerces to `[]` whenever `appAccess` is false at `src/features/people/actions.ts:426`. The transaction syncs that coerced value at `:462`. `syncTeamMemberRoles` archives active v1 roles absent from the desired set at `:199`. Reserved roles (`staff`, `viewer`) are intentionally untouched because the archive filter limits to `V1_TEAM_ROLES` at `:204`.

### Trigger-won path on `createPerson` — verified

On zero-row claim it reads `contacts.user_id` at `:373`; if it equals the just-created auth id, it proceeds to success. Compensation only fires when the link points elsewhere.

### Other admin-mint paths

Only two in the codebase: `syncAuthMetadata` (new) at `actions.ts:271` and the legacy `applyRoleSet` at `src/features/auth/actions.ts:419`. Both go through `auth.admin.updateUserById({ app_metadata: { role } })`. No other mint paths. The legacy one is scheduled for deletion in Phase 4.

### Test fidelity nit (carry-forward)

The stale-role coercion test queues an empty existing role set at `actions.test.ts:408`, so it doesn't observe the archive branch at `actions.ts:207`. Production code is correct (Codex confirmed by code reading); this is only a test-strength gap. Closing it would mean queuing a non-empty existing role set + asserting the archive UPDATE was called. Trivial follow-up.
