## Codex Adversarial Review

### High

1. **Setter actions can silently lose edits during lifecycle races.** `setQuoteInputs` and `setQuoteTax` read status, compute patch, run guarded `UPDATE ... WHERE id=? AND status='draft'`, then return `{ok:true}` without checking row count. Under READ COMMITTED, a concurrent `sendQuote` between SELECT and UPDATE → UPDATE affects 0 rows but action reports success.

### Medium

1. `inputs` JSON accepts and persists arbitrary extra keys. Spread merge means `{"__proto__":{...},"blob":"…",audienceSize:500}` lands in the `quotes.inputs` jsonb after validation passes the known fields.
2. Record-retrieval range pricing not enforced server-side. UI pills constrain to 0/100/200/300/400, but a direct Server Action can submit `recordRetrievalAmount: 9999999`.
3. `setQuoteTax` bypasses tax cap. `parseTax` accepts up to `Number.MAX_VALUE`; only `computeQuote` enforces `MAX_DOLLARS=9_999_999`. Direct `setQuoteTax(tax='999999999999')` hits a numeric(12,2) overflow at the DB instead of a friendly validation error.
4. Dealer archive race in `setQuoteDealer` and `createQuote`. Active-dealer SELECT runs separately from the write — concurrent archive could attach a quote to a freshly-archived dealer.

### Low / Nit / False positives
None new.

## Action Required

### Must Fix (blockers)
None from static or browser.

### Should Fix (Codex high/critical)
1. Add `.returning({ id })` + miss-classification to `setQuoteInputs` and `setQuoteTax`.

### Consider (Codex medium/low)
1. Canonicalize `parseQuoteInputs` (whitelist fields, discard unknown keys).
2. Enforce catalog `[unitPriceMin, unitPriceMax]` for `range` items in `computeQuote.emitVariable`.
3. Cap `parseTax` at `MAX_DOLLARS` (export the constant from pricing.ts).
4. Wrap `createQuote` + `setQuoteDealer` in a transaction with `.for('update')` on the dealer SELECT.
