## Codex Adversarial Review

(Captured pre-app-chip-removal — findings are listed verbatim. The medium on the chip's banned-user semantics is now moot because the chip itself was removed in this same Phase 2 review pass.)

**Critical:** None.
**High:** None.

**Medium**

1. **Sortable headers are mouse-only** — `src/components/ui/data-table.tsx:113-117`
   Sorting is attached directly to `<th onClick>`, with no focusable control or keyboard handler, so keyboard users cannot sort the table. Suggested fix: render a `<button type="button">` inside sortable headers and put `getToggleSortingHandler()` on that button.

2. ~~**`app` chip overstates disabled/banned app access**~~ — `people-columns.tsx:87-94`
   Resolved: the `app` chip was removed in this Phase 2 review pass per the new "App is implicit, not signaled" convention. No code-level guard required.

**Low**

3. **Advertised `globalFilterFn` prop is ignored** — `src/components/ui/data-table.tsx:43-44`
   `DataTableProps` exposes `globalFilterFn`, but the component never destructures or passes it into `useReactTable`. No current Phase 2 UI uses it, but Phase 4's planned cross-column search will silently fall back to TanStack defaults. Suggested fix: destructure and pass to `useReactTable({ globalFilterFn })`.

4. **Last-sign-in visibility is population-based, not visible-row-based** — `src/features/people/people-admin.tsx:46-51`
   The plan says hide `Last sign-in` when no visible row has app access, but the memo checks the full `people` array. Not visible until filters land (Phase 4); after Phase 4 a filtered customer-only view can still show the column because another hidden row has app access. Suggested fix: tie visibility to filtered data, or accept the population-level behavior and update the plan/copy.

5. **Empty-state `colSpan` counts hidden columns** — `src/components/ui/data-table.tsx:142-146`
   When `lastSignInAt` is hidden, the empty row still uses `getAllLeafColumns().length`. Browsers tolerate this, but it is structurally wrong. Suggested fix: use `table.getVisibleLeafColumns().length`.

**Not an issue (false positives ruled out)**

- `AdminPersonRow` literals outside `queries.test.ts` — none found; tests already updated.
- Extra `loadAdminPeople` consumers — only `/admin/people` page; row passed straight into `PeopleAdmin`.
- Raw first/last empty strings — both client- and server-side validation block empty submissions.
- Dropped subtitle/subhead copy has no runtime references.
- Column definitions are not recreated in render scope (module-level `buildPeopleColumns`).
- Accessor usage is coherent (explicit `id` everywhere `accessorFn` is used).
- Small-population pagination handles empty/single-page tables correctly.

## Coverage Caveats

- Codex review was captured before the App-access-checkbox removal pass; that change re-shaped `PersonForm` (derived `wantsAppAccess` instead of state). Mechanically the change is small (no new branches; existing Server Action contract unchanged), but a re-run of Codex against the final diff would close the loop on the `disabled={!appAccess}` removal and the new pre-submit validation copy. Tagged "with caveats" for this reason rather than full PASS.
- Phase 3 confirm dialogs (window.confirm) and the auto-uncheck task have been re-scoped: the auto-uncheck item is now obsolete (no `App access` toggle to flip); the disable-app-access confirm trigger has changed to "all roles unticked on a person who currently has app access." Plan updated.
- The legacy "coach without auth user" state (Adam Godin) is now editable through the new dialog — opening Edit on him with `coach` ticked will provision the missing auth user on save. Acceptable migration path; no backfill scoped.
