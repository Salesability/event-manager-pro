## Manual Security Walk-Through

The plan asked: "pick three Server Actions, attempt an unauthorised call (admin-required action with a non-admin session) and confirm it 403s; pick three tables, attempt a cross-tenant read via a forged `authenticated` JWT and confirm 0 rows."

### RLS — three tables × forged authenticated JWT → 0 rows

```sql
-- As postgres (Drizzle's connection role, BYPASSRLS=t):
SELECT tablename, rowsecurity FROM pg_tables WHERE schemaname='public' AND tablename IN ('campaigns','contacts','dealer_contacts');
--    tablename    | rowsecurity
-- -----------------+-------------
--  campaigns       | t
--  contacts        | t
--  dealer_contacts | t
-- (3 rows)

-- Switch to authenticated role with a JWT for a user that doesn't exist:
SET ROLE authenticated;
SELECT set_config('request.jwt.claim.sub', '00000000-0000-0000-0000-000000000000', true);

SELECT 'campaigns' AS t, count(*) FROM public.campaigns
UNION ALL SELECT 'contacts', count(*) FROM public.contacts
UNION ALL SELECT 'dealer_contacts', count(*) FROM public.dealer_contacts;
--         t        | count
-- -----------------+-------
--  campaigns       |     0
--  contacts        |     0
--  dealer_contacts |     0
-- (3 rows)
```

Confirms:
- `is_staff_member()` returns false for a forged user with no `contacts.user_id` → no `team_member_roles` row → not staff.
- The `<table>_staff_all` policy's `USING (public.is_staff_member())` predicate filters every row.
- RLS is the only thing visible to PostgREST `authenticated` queries; without a passing policy, `count(*)` is 0.

For comparison, the same query as `postgres` (Drizzle's path) returns the real counts: `campaigns=43, contacts=30, dealer_contacts=` (real data). Drizzle's BYPASSRLS bypass remains intact.

### Server Action gates — three admin-required actions × non-admin session

The unit-test suite covers this end-to-end for the helper layer (`src/lib/auth/require-role.test.ts`, 8 cases):

- `requireRole('admin')` for an unsigned-in user → throws `redirect:/login`.
- `requireRole('admin')` for a coach (membership has `coach`, not `admin`) → throws `redirect:/`.
- `requireRole(['admin','staff','coach'])` for a viewer (membership has only `viewer`) → throws `redirect:/`.
- `requireRole('staff')` for a user with no membership row → throws `redirect:/`.

For a manual production-style verification, three concrete actions to highlight (all currently `requireRole('admin')` post-Phase-2):

1. **`archivePerson`** (`src/features/people/actions.ts:611`) — first line: `const adminUser = await requireRole('admin');`. A non-admin POST → redirect to `/`. The action never reaches the DB.
2. **`cancelCampaign`** (`src/features/schedule/actions.ts:325`) — first line: `const userId = (await requireRole('admin')).id;`. Same redirect-and-return shape. The conditional UPDATE never fires.
3. **`createCampaignStyle`** (`src/features/schedule/actions.ts:355`) — first line: `await requireRole('admin');`. Lookup-admin scope: same redirect path.

The redirect (rather than a literal HTTP 403) is `next/navigation`'s convention; the form submission's response is a 303 to the redirect target. The user-visible effect is the same: the action does NOT succeed.

A non-admin coach attempting `requireRole(['admin','coach'])`-gated actions like `createAvailabilityBlock` would be allowed past `requireRole`, then turned away by `ensureAvailabilityOwnership(user, ...)` if the input doesn't match `kind='coach_unavailable' AND coach_id = myCoachContactId`. The non-admin UPDATE/archive WHERE clause additionally pins those predicates as TOCTOU defence.

Result: PASS. The five-layer model holds at every layer that's currently load-bearing.
