## Codex Adversarial Review

Verbatim from `codex:codex-rescue`:

**Critical / High:** None.

**Medium**

> `src/features/dealers/dealers-admin.tsx:38` — archive confirm says "any contact links to this dealership stay readable." Observation: `archiveDealer` only sets `dealers.archivedAt` and does not cascade or hard-delete links (`src/features/schedule/actions.ts:271`), but the People query hides dealer links when the dealer is archived via `isNull(dealers.archivedAt)` (`src/features/people/queries.ts:112`). Rows remain in DB, but they are not "readable" in the active People admin surface. Why it matters: the confirm text misleads admins about visible contact-link behavior. Suggested fix: rephrase to avoid promising visibility, e.g. "Existing campaigns keep their reference. Contact relationship rows are preserved, but archived dealers no longer appear in active dealer pickers."

**Resolution:** Fixed in-eval. Verified Codex's claim by grepping `src/features/people/queries.ts` — line 116 confirmed `isNull(dealers.archivedAt)` filters archived dealers out of the People admin's contact-link rendering. Rephrased the message to: *"Archive ${dealer.name}? Existing campaigns keep their reference. Contact-link rows are preserved in history but the dealer disappears from the People page and Dealer pickers. Continue?"* This is factually accurate without overpromising. Tsc + lint re-run clean.

**Low**

> `docs/designs/0027-lists-to-dealers/plan.md:102` — checklist says DealerForm has the "same `useEffect` toast/router-refresh wiring" as PersonForm. Observation: success wiring matches, but error wiring differs: PersonForm refreshes after error (`people-admin.tsx:402`), DealerForm only toasts (`dealer-form.tsx:44`). Why it matters: mostly plan accuracy; no dealer-action error path commits partial data and needs a refresh.

**Resolution:** Acknowledged, no code change. Dealer error paths are pure validation + ConflictError surfacing — none commit partial DB state, so the post-error refresh PersonForm does (which exists for the auth-side partial-success edge case in user-create) doesn't apply here. Plan checklist wording remains as a structural-parity claim, not a literal "every line matches" claim.

**Explicit non-issues** (Codex confirmed)

- Wire format is correct: form posts `name`, `contactFirst`, `contactLast`, `contactEmail`, `contactPhone`, `address`, edit-only `id`; actions read those exact names at `src/features/schedule/actions.ts:59` and `:137`.
- Trimming + validation are server-owned via `field()` validator.
- Hidden edit `id` is type-compatible with `Number()` positive-integer validation.
- `Dialog.Close` renders as `type="button"` per Radix internals — Cancel doesn't accidentally submit.
- Dialog state resets on close: Add form mounts only while `addOpen` is true; Edit form mounts only while `editing` is non-null. Resets `useActionState`.
- Focus behavior matches PersonForm: `autoFocus` on first input + Radix Content's FocusScope mount delegation.
- Dead-code removal complete: `rg` finds no remaining references to `AddDealerButton`, `DealerRowActions`, the old route-form import path, or `list-actions` in `src/`.

**Codex ship verdict:** "Ship after fixing the misleading archive confirm copy, or accept that as a small follow-up if this patch needs to land now." → Fixed in-eval, no carry-forward.

## Coverage Caveats

- Edit-row click hit the same strict-mode collision PersonForm hits (26 ambiguous `Edit` buttons). Source-trace + Codex source review covered the Edit path; a Playwright-style harness that targets row index would be needed to drive the actual Edit dialog. Not introduced by this phase.
- Form submission was not exercised in smoke (read-only discipline — would create/modify a real dealer row). Wire-format verified by Codex source review against `createDealer`/`updateDealer` parsers. The actions are unchanged from Phase 2 — the form is wire-compatible with the deleted form, which was shipping fine pre-rewrite.
