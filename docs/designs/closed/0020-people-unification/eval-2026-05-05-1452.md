## Codex Adversarial Review

### High â€” blocks commit

**Concurrent offâ†’on `updatePerson` can mint an unlinked auth user, possibly admin.**

- File: `src/features/people/actions.ts:442` (the offâ†’on branch).
- Repro: Two admins both load the same contact (Tilley), both submit `appAccess=1` with different emails. Both `auth.admin.createUser` calls succeed (Supabase is happy to create two distinct users, two emails). The first wins the guarded `UPDATE contacts SET user_id = ... WHERE id = ? AND user_id IS NULL`; the second updates zero rows but **still calls `syncAuthMetadata`** â€” and if `roles` includes `admin`, the second auth user ends up with `app_metadata.role = 'admin'` while having no `team_member_roles` link.
- Why it matters: `requireAdmin` trusts only `app_metadata.role` (`src/lib/auth/require-admin.ts:5`), so this can mint an unlinked admin account that passes the gate. The orphan account isn't visible in `/admin/users` (no contact link), but it can sign in.
- Fix sketch: make the guarded update `RETURNING { id }`; on zero rows, treat as conflict, ban + clear `app_metadata.role` on the just-created auth user (compensating delete), and return a "raced; refresh" error. Same pattern as the 0018 TOCTOU fix on `linkUserToContact`.

### Medium â€” should fix before ship

**Roles + `appAccess=off` survives a stale UI submission.**

- File: `src/features/people/actions.ts:416`.
- Repro: a linked coach submits with `appAccess=off` but the UI form still includes `roles=coach`. The server keeps/restores the active role rows in `syncTeamMemberRoles`, then bans the auth user. Result: the contact has `team_member_roles(role='coach')` but the linked auth user is banned â€” internally inconsistent, contradicts the old `deactivateUser` shape.
- Fix sketch: when `!appAccess`, force `roles = []` before the role-set sync (don't reject â€” the UI may not have updated; coerce silently or with a clear server-side note). One line.

### Low

1. **`appAccess === '1'` parses only the literal `'1'`.** `src/features/people/actions.ts:293, :386`. Browser checkbox defaults like `'on'`, `'true'`, or trailing whitespace silently become false. Centralize a `parseBoolField` helper.
2. **`dealerLinks` parser accepts `100:staff:extra`.** `src/features/people/actions.ts:78`. `entry.split(':')` returns 3 parts; the third is silently dropped. Reject anything that isn't exactly two segments.

### Confirmed-OK / adversarial questions answered

- **Transaction boundary** â€” partial-success message on `auth.admin.createUser` failure is clear; no auth orphan created when `createUser` fails. Retry via `updatePerson` offâ†’on works in the non-racy case.
- **Conditional `contacts.user_id` UPDATE** â€” correct for the "trigger won the race" path during create; broken in the concurrent-update path (the High above).
- **Self-archive guard** â€” multi-link is impossible per `contacts_user_id_unique` (`drizzle/0000_cute_ser_duncan.sql:216`).
- **`syncDealerLinks`** â€” restore-archived-row-before-insert correctly avoids the unique-index conflict (`dealer_contacts_dealer_contact_role_unique` is a full unique index, not partial).
- **`syncTeamMemberRoles`** â€” same: full unique index, restore-before-insert handles it. Leaves `staff` / `viewer` rows alone (correct for v1).
- **Identifier flow** â€” archived rows accumulate intentionally, matching `schedule/actions.ts` pattern.
- **`'use server'`** present; only async `ActionResult` exports.
- **Test mock** â€” predicate-blind by design (carried-forward Low). `tableName` introspection via Drizzle's internal `Symbol(drizzle:Name)` is stable across recent versions but a future Drizzle bump could fall back to `'unknown'` and silently weaken table-mutation assertions.
