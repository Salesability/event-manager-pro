## Codex Adversarial Review

> **Critical**
>
> None found.
>
> **High**
>
> Authenticated users can replay real email sends without limits — `src/features/email/actions.ts:36`, `src/features/email/actions.ts:75`, `src/features/email/actions.ts:111`
> This WILL let any signed-in user who can call the Server Actions submit arbitrary `campaignId` / `coachId` values and repeatedly trigger sends to DB-derived recipients at `src/features/email/actions.ts:71`, `src/features/email/actions.ts:107`, and `src/features/email/actions.ts:140`. Existing schedule actions are also login-only (`src/features/schedule/actions.ts:389`, `src/features/schedule/actions.ts:414`, `src/features/schedule/actions.ts:441`), and RBAC is explicitly not shipped yet (`docs/wiki/auth.md:96`), but email is an external side effect. Exploit: a compromised/pre-created account loops 10k requests over known IDs and floods clients/coaches. Fix: add server-side rate limits, audit/idempotency rows keyed by user + recipient + campaign/template, and role checks before send.
>
> Share-link email trusts request Host — `src/features/email/actions.ts:21`
> `siteUrl()` builds the emailed URL from `host` and `x-forwarded-proto` headers (`src/features/email/actions.ts:23`-`26`), then sends it to coaches (`src/features/email/actions.ts:128`, `src/lib/email/templates.ts:172`). If the runtime/proxy accepts an attacker-supplied Host on a valid authenticated action request, the coach receives a legitimate-looking email pointing at an attacker domain. Fix: use a canonical `APP_URL`/`NEXT_PUBLIC_SITE_URL` allowlisted origin, not request headers.
>
> **Medium**
>
> Cancelled/completed campaigns can still get "confirmation" emails — `src/features/email/actions.ts:43`
> Both campaign email actions load any campaign by numeric id and never check `campaign.status` before sending (`src/features/email/actions.ts:43`-`49`, `src/features/email/actions.ts:82`-`88`). This WILL allow replaying booking/assignment confirmations for cancelled or completed events via direct action calls, even if the UI usually hides cancelled rows. Fix: reject anything except the intended lifecycle state, likely `booked`.
>
> Production/dev redirect has dangerous config combinations — `src/lib/email/send.ts:23`
> The matrix is: `APP_ENV=production` always sends real; `APP_ENV` unset + `NODE_ENV=production` sends real; `APP_ENV` unset + `NODE_ENV!=production` redirects when `EMAIL_DEV_TO` is set; any non-production `APP_ENV` redirects regardless of `NODE_ENV` (`src/lib/email/send.ts:23`-`38`). So there is a path where a production deploy silently sends to `EMAIL_DEV_TO`: live env with `APP_ENV` unset and `NODE_ENV` not exactly `production`, or `APP_ENV=development`/typo. Fix: require explicit `APP_ENV=production` for configured production deploys, or invert to an explicit `EMAIL_FORCE_DEV_REDIRECT=true`.
>
> Header/control-character sanitation is delegated entirely to Resend API — `src/lib/email/send.ts:34`
> Subjects include DB text (`dealerName`) at `src/lib/email/templates.ts:55` and `src/lib/email/templates.ts:117`, and dev mode prefixes raw `to` into the subject at `src/lib/email/send.ts:37`. The installed SDK does not strip CR/LF: it maps `subject` unchanged (`node_modules/resend/dist/index.mjs:187`-`198`) and JSON-posts it unchanged (`node_modules/resend/dist/index.mjs:1127`-`1134`). Impact depends on Resend API validation, but the app should reject `\r`/`\n` in `to`, `from`, `replyTo`, and `subject` before calling the SDK.
>
> **Low**
>
> Share-link send loads every campaign then filters in JS — `src/features/email/actions.ts:122`
> This is suboptimal, not a correctness break today. `sendCoachShareLinkEmail()` calls full-table `loadCampaigns()` (`src/features/schedule/queries.ts:329`-`361`) and filters at `src/features/email/actions.ts:124`-`126`; at scale this adds avoidable latency and memory. Fix with a targeted `loadUpcomingCampaignsForCoach(coachId, today)` query.
>
> **Checked and NOT a problem**
>
> - Resend v6.12.2 `replyTo` shape is correct: public type accepts `replyTo` (`node_modules/resend/dist/index.d.mts:547`-`551`) and SDK maps it to API `reply_to` (`node_modules/resend/dist/index.mjs:187`-`198`).
> - `requireUserEmail()` does not allow an empty reply-to; falsy email returns an error (`src/features/email/actions.ts:15`-`18`), and `sendEmail()` omits falsy `replyTo` (`src/lib/email/send.ts:46`).
> - `notes` is not in the subject; it only enters the text body (`src/lib/email/templates.ts:61`, `src/lib/email/templates.ts:118`).
> - Date formatting avoids local DST drift by forcing UTC (`src/lib/email/templates.ts:5`-`20`).
> - Campaign confirmation's extra `loadCampaign` + optional `loadCoach` round-trip is acceptable for one send (`src/features/email/actions.ts:43`, `src/features/email/actions.ts:49`).

## Action Required

### Must Fix (blockers)

_None — all static + browser checks clean._

### Should Fix (Codex high)

1. **Auth/abuse hardening on email actions** (`src/features/email/actions.ts:36,75,111`). Minimum viable: per-user rate limit (e.g. 10 sends/min), idempotency on `{userId, campaignId, template}` so double-clicks don't double-send, and a `messages_sent` audit row (already on the deferred list in the chunk plan — promote it). Role gating can wait until RBAC lands.
2. **Canonical site URL for share links** (`src/features/email/actions.ts:21-26`). Add `NEXT_PUBLIC_APP_URL` (or `APP_URL`) env var, use it in `siteUrl()`, fall back to request headers only when unset (i.e. local dev). Coaches' inboxes should never receive a host they didn't expect.

### Consider (Codex medium/low)

1. **Status-gate the campaign sends** (`src/features/email/actions.ts:43-49,82-88`) — refuse `cancelled`/`completed`. Tiny check, prevents weird replay.
2. **Tighten the prod/dev matrix** (`src/lib/email/send.ts:23-38`) — invert to `EMAIL_FORCE_DEV_REDIRECT=true` or require `APP_ENV=production` explicitly. Today: a Vercel/GCP deploy that forgets to set `APP_ENV` but still has `NODE_ENV=production` is fine; the worry is non-standard runtimes where `NODE_ENV` ≠ `production` but `APP_ENV` is also unset → silent redirect.
3. **CR/LF sanitation on outbound fields** (`src/lib/email/send.ts:34,37`, `templates.ts:55,117`) — strip `\r\n` from `subject`, `to`, `replyTo` before handing to Resend. Belt-and-suspenders against header injection if Resend ever loosens validation.
4. **Targeted query for share-link** (`src/features/email/actions.ts:122`) — replace `loadCampaigns()` + JS filter with a `loadUpcomingCampaignsForCoach(coachId, isoDate)` query. Defer until campaign count crosses ~5k.
