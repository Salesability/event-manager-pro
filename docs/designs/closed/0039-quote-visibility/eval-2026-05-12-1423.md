## Codex Adversarial Review

### Pass 1 — 2 Medium (self-resolved)

**Medium 1 — Archived dealer rows now get an unconditional `/dealerships/${id}` link.**

> The detail loader uses `loadDealer()`, which filters `archived_at IS NULL`. Since `/dealerships` intentionally loads archived rows, filtering to the Archived pill and clicking a dealer name routes admins to a guaranteed 404.

**Medium 2 — `Number(idParam)` + `Number.isInteger` accepts non-decimal route params.**

> `1e10`, `0x10`, `1.0` all pass `Number.isInteger` after `Number()` coercion. The intent is a "positive int path segment" and this admits noncanonical deep links plus unsafe precision behavior for bigint-backed ids.

### Fixes applied (direct edits)

- `src/features/dealers/dealers-columns.tsx` — Name cell now branches on `archivedAt`: archived rows render `<span className="font-medium text-stone-500">{d.name}</span>` (plain text), non-archived rows keep the `<Link>` to `/dealerships/<id>`. TanStack `accessorKey: 'name'` is unchanged, so sort/filter behaviour is unaffected (both read the underlying row, not the cell render).
- `src/app/(app)/dealerships/[id]/page.tsx` + `src/app/(app)/quotes/[id]/page.tsx` — new local helper `parsePositiveIntPathSegment(v: string): number | null` declared in each page. Regex `/^\d+$/` (rejects `1e10` / `0x10` / `1.0` / `-1` / `+1` / leading-whitespace), `Number.isInteger`, `>0`, `<= Number.MAX_SAFE_INTEGER` (bigint upper bound). Same helper in both files — three-line dup is intentional per repo "three similar lines is better than a premature abstraction" convention; a shared util becomes the right home if a third dynamic-id route ever appears.

### Pass 2 — Verify

Codex pass-2 returned **1 Medium (false positive)** + 2 confirmed false-positive entries on the original findings.

**Pass-2 Medium claim:**

> The parser still accepts `1\n`: in JavaScript, `$` can match before a final line terminator, so `/^\d+$/` succeeds and `Number("1\n")` returns `1`.

**Verdict on the claim: FALSE POSITIVE.**

Verified with `node -e "console.log(/^\\d+\$/.test('1\\n'))"`:
- `/^\d+$/.test('1\n')` → **`false`** in Node 20+.
- Per ECMAScript spec: when the `multiline` flag is off (default), `$` matches end-of-input only — NOT before a trailing `\n`. The MDN "Boundary-type assertions" page confirms this.
- Independent of regex behaviour, Next.js dynamic-route `params` values are URL-decoded path segments; the router rejects unencoded newlines in URL paths anyway.

Both lines of defense reject the alleged bypass. No fix needed.

**Pass-2 false-positives confirmed (originals closed):**

> The archived-dealer 404 finding appears closed: archived rows render a plain `<span>` and only non-archived rows render the `/dealerships/${d.id}` Link. Sorting remains tied to `accessorKey: 'name'` with `enableSorting: true`.

> The duplicate helper comment blocks are consistent across both pages, and the two local declarations are acceptable as intentional duplication.
