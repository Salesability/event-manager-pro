## Codex Adversarial Review

**Critical**

None.

**High**

1. `drizzle/0008_bright_lockjaw.sql:2` and `tests/integration/rls.test.ts:27-39`

   What's wrong: `master_service_agreements` is created as a new public domain table, but the migration does not enable RLS or add the standard service-role/staff policies, and the RLS baseline test list omits the table. This conflicts with the existing baseline: `docs/wiki/auth.md:146` says "Row Level Security on every public table," and `drizzle/0003_enable_rls.sql:1-3` says Phase 1 enabled RLS on every public domain table.

   Why it matters: This table will store signed-contract metadata and signed PDF storage keys. "No rows yet" makes the immediate exposure low, but the schema is now live and 7.2 can start writing sensitive rows unless another migration lands first.

   Suggested fix: Add RLS for `public.master_service_agreements` before any write path ships, preferably in this migration if still unapplied, otherwise in the next migration. Mirror the `service_role_all` and `authenticated staff_all` pattern from `drizzle/0003_enable_rls.sql`, and add `master_service_agreements` to `RLS_TABLES` in `tests/integration/rls.test.ts`.

**Medium**

1. `tests/integration/master-service-agreements-schema.test.ts:55-59`

   What's wrong: The reachability test asserts the table has zero rows: `expect(rows).toEqual([])`. That is not a schema invariant; it will fail as soon as 7.2 or manual smoke data inserts a valid MSA row.

   Why it matters: This turns a thin schema test into an environment-state test. Future valid data will create false negatives in local or shared integration runs.

   Suggested fix: Change the assertion to prove the table can be selected from without assuming emptiness, e.g. query `select 1 from public.master_service_agreements limit 1` and assert the query resolves, or inspect `information_schema.columns`.

**Low**

1. `docs/wiki/data-model.md:137-150`

   What's wrong: The current-state data model table still omits `master_service_agreements`, even though `docs/designs/0037-commercial-spine-msa/plan.md:54` explicitly says "MSA section lands in 0037 Phase 2." The new table is present in schema/migration, but the wiki remains stale.

   Why it matters: The repo's doc convention says `docs/wiki/` describes current state. After this phase, readers looking at the data model will not see the MSA table or its dealer relationship.

   Suggested fix: Add a concise `master_service_agreements` entry to the table list and relationships/audit sections. Defer the `quotes` section until 0026 Phase 2, as the plan already says.

2. `tests/integration/master-service-agreements-schema.test.ts:4-7`

   What's wrong: The test comment says it confirms "the unique constraint shape on `dealer_id` + `status` is reachable," but the schema only creates a non-unique index at `src/lib/db/schema/master-service-agreements.ts:31-33`. There is no unique constraint and no test for one.

   Why it matters: This misstates the integrity boundary and can mislead later reviewers into thinking "one active MSA per Client" is database-enforced.

   Suggested fix: Update the comment to say it checks table reachability and enum values only. If database enforcement of one active MSA is desired later, add an explicit partial unique index in a planned migration.

**Non-Issues Considered**

- `templateVersion` as `NOT NULL`: defensible because the plan's OQ #6 says the version is hardcoded server-side when signing starts, and pending rows are created as part of that signing flow.
- Timestamp types: all date-time columns in the schema and migration use `timestamp with time zone`, matching the repo convention.
- String types: all textual columns use `text`, not `varchar`.
- FK directions and deletes: `dealer_id -> dealers.id ON DELETE restrict`; actor FKs point to `auth.users ON DELETE set null`, matching `actors`.
- FK indexes: `dealer_id`, `created_by_id`, and `updated_by_id` are indexed; planned `(dealer_id, status)` and `expires_at` indexes are present.
- Spurious auth schema gotcha: `drizzle/0008_bright_lockjaw.sql` contains no `CREATE SCHEMA "auth"` or `CREATE TABLE "auth"."users"` statements.
- Migration destructiveness: `0008` is create-only plus constraints/indexes; no destructive operations.
- Journal bump: in-repo precedent exists for manual `when` bumps. Original generated `when` (`1778532082809`) sorted before `0007` (`1778677200000`), so drizzle-kit skipped applying `0008`. Bumped to `1778763600000` (= `0007` + 1 day, matching the cadence of prior entries) before reapplying. Verified the table now exists via `psql \d public.master_service_agreements`.
- Scope creep: no Server Actions or Dropbox Sign flows were added.
- Cross-plan drift: 0025/0026/0035 still align on accepted Quote as contract, no orders table, `campaigns.acceptedQuoteId -> quotes.id`, and MSA gating on Send.
