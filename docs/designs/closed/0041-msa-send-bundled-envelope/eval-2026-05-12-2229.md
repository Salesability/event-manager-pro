## Codex Adversarial Review

### High

**Duplicate Dropbox Sign envelopes can be sent under concurrent `sendMsaEnvelope` calls** — `src/features/msa/actions.ts:194-209` + `:290-345`. Two callers can both read `dropboxSignDocumentId = null`, both upload + post envelopes, and only one `signatureRequestId` is persisted. The loser's envelope still reaches the client, but its webhook has no MSA row to correlate to and no `msa.sent` audit is written for it. The atomic guarded UPDATE closes the *DB* race, not the *external-API* race. Suggested fix shape: claim the send before external side effects (e.g. a `send_in_progress` marker or per-MSA advisory lock), and classify guarded-update misses instead of treating every miss as idempotent success.

### Medium

1. **`testMode` env-trim asymmetry** — `src/lib/dropbox-sign/client.ts:83` uses `process.env.APP_ENV !== 'production'` without trim+lowercase. `email/send.ts` *does* trim+lowercase. So `APP_ENV=" production "` real-sends email but sends a Dropbox Sign test-mode envelope; if the webhook then fires for the test envelope, the MSA could flip to `active` against a test-mode artifact. Fix shape: a shared env-normalization helper, or refuse e-sig sends unless production mode is explicit.

2. **First-draft-Quote picker is dealer-scoped, not coach-scoped** — `firstDraftQuoteIdForDealer` returns the earliest draft Quote for a dealer regardless of which coach created it. In multi-coach dealers, the bundled envelope could carry a different coach's draft. Fix shape: filter by `createdById` if that's the ownership proxy, or surface an explicit Quote picker in the dialog.

3. **Step-2 send failures leave a pending MSA with no UI retry path** — if `createMsaDraft` succeeds but `sendMsaEnvelope` fails (Dropbox down, GCS upload error), the panel flips to the `msa ?` branch and hides the create trigger; the existing-MSA guard on `createMsaDraft` then refuses a second attempt. The action *is* idempotent on `dropboxSignDocumentId IS NULL`, so a "Resend envelope" button would work — but the UI doesn't render one today. Fix shape: render a resend button when `pending && !dropboxSignDocumentId`, or have `sendMsaEnvelope` auto-retry the create-cleanup on pre-envelope failure.

### Low

**Tests don't exercise the real send race** — the predicate-blind DB mock doesn't model two concurrent `sendMsaEnvelope` calls both reaching `sendSignatureRequest`. A controlled promise-barrier test would prove only one caller can post. Lower priority than the High because the race is real but operationally rare (it requires two simultaneous UI clicks against the same MSA, which the current single-flow dialog doesn't surface).

### Explicitly not an issue

- HMAC length-check-first leak — `computed` is SHA-256 hex from `.digest('hex')`, always 64 chars; the only fact revealed is whether the *caller-supplied* hash length is 64. Server-side secret is not in the disclosed envelope.
- HMAC verification ordering — runs before any DB read or mutation in the route handler.
- `createMsaDraft` double-create race — closed by the dealer `FOR UPDATE` lock + same-tx existing-MSA check.
- Upload-before-API ordering — draft PDF upload is awaited + error-checked before `sendSignatureRequest`, so no silent-upload-failure path.
- `currentMsaTemplateVersion()` env access — request-time, not module-load-time.
- `'use server'` file only exports the two gated async actions.
- Audit-log writer with `actorUserId: null` / `actorRole: 'system'` — consistent with schema (`actor_user_id` nullable, `actor_role` text-unconstrained).
- PDF prose interpolation — drawn as glyphs by pdf-lib's `drawText`, not parsed; no template/code injection.
- Signed-webhook deleted-row orphan window — bounded by `master_service_agreements.dealer_id` being `ON DELETE RESTRICT` and no app delete path for MSA rows today.
