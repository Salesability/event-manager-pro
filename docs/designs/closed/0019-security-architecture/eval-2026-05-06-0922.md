## Codex Adversarial Review

**Critical:** None.

**High:**

1. **Coaches can mutate global and other-coach availability blocks.** `src/features/schedule/actions.ts:540,559,580`. The `requireRole(['admin','coach'])` gate I applied per the plan example doesn't enforce "their own block" ownership — `parseAvailabilityInput` permits `statutory_holiday` / `company_closure` with no coach id, and update/archive only filter by block id + unarchived state. **Status: deferred.** The plan explicitly notes "the change is a 3-character edit per call site, not a refactor"; row-level ownership checks are out of scope. Worth a dedicated follow-up chunk before any non-admin coach is provisioned in production. The current production user set is admin-only (David), so the practical risk is low today.

2. **Four mutating Server Actions still rely on authn-only `requireUserId()`** (createDealer, updateDealer, createCampaign, updateCampaign). Codex's argument: Server Actions are public API endpoints; the layout gate doesn't defend a direct POST. **Status: fixed in-eval.** Tightened to `requireRole(['admin','staff','coach'])` (viewer excluded since they shouldn't mutate). The now-orphan `requireUserId()` private helper deleted; `getUser`/`redirect` imports cleaned up. Re-ran tests + tsc clean.

**Medium:**

3. **Admin fast-path is only as fresh as `app_metadata.role`.** `src/lib/auth/require-role.ts:28`. Admin bypasses `loadCurrentMembership()`, so a stale `app_metadata.role='admin'` after a failed `syncAuthMetadata()` (which returns a warning *after* the DB role-removal commits) would preserve admin until the next role write or a manual fix. **Status: known follow-up.** Listed in `CURRENT.md` Parked as "0018 follow-up: createUser partial-success after auth-side success." Not introduced by this phase; pre-existing 0018 carry-forward.

**Low:**

4. **People action tests are predicate-blind for `requireRole` arguments.** `src/features/people/actions.test.ts:101`. The mock resolves regardless of which role list production passes. **Status: parked.** Same predicate-blindness theme already noted in 0018's Codex Low list.

5. **Narrow TOCTOU between role check and DB write.** Theoretical: a role revocation that lands between `requireRole(...)` and the SQL UPDATE wouldn't be honored. Drizzle bypasses RLS so there's no DB-side backstop. Practical concern is near-zero (staff churn is rare; revocations are admin-initiated and rare). **Status: noted; no action.**

**Explicitly cleared (not issues):**
- No leftover `requireAdmin` callers under `src/`. The async wrapper is fully retired; only `isAdmin(user)` predicate remains.
- `loadCurrentMembership()` cache is request-scoped via React `cache()`, not cross-request stale.
- Email unauth redirects propagate correctly; no caller swallows the throw from `redirect()`.
- `/admin/people` has layered admin gates (middleware + page-level `requireRole('admin')`).
