## Codex Adversarial Review

Codex raised 2 Lows, no Critical / High / Medium. Both fixed in-eval:

**Low #1 — Duplicated staff-app role list can drift.**
File: `src/lib/auth/capabilities.ts:73`. The original implementation hard-coded `admin || staff || coach || viewer` inline; the canonical list lives in `STAFF_APP_ROLES` at `load-team-membership.ts:16`. A future change to the constant would silently desync the predicate.
**Resolution:** extracted `STAFF_APP_ROLES`, `isStaffAppRole`, and `TeamMemberRole` to `src/lib/auth/team-roles.ts` (no `'server-only'` directive — client-safe constants module). `load-team-membership.ts` now re-exports those symbols so every existing import path (route handler, route test mock, require-staff-access.ts) keeps working unchanged. `capabilities.ts` imports `isStaffAppRole` directly and calls `profile.roles.some(isStaffAppRole)` — single source of truth.

**Low #2 — Multi-role hybrid behavior not pinned by tests.**
The Phase 1 test suite covered single-role profiles for each role but not hybrids like `['coach', 'dealer']`. The team_member_roles schema only enforces `(contactId, role)` uniqueness; the Person dialog exposes Admin/Coach/Dealer as independent checkboxes. So hybrids are reachable in production.
**Resolution:** added a `hybrid-role profiles` describe block with 8 assertions: `coach+dealer` → grants `reports:view`/`availability:edit`/`app:access`, denies `admin:access`; `staff+dealer` → grants `app:access`, denies `reports:view`; `viewer+dealer` → grants `app:access`, denies `availability:edit`. Pins "any matching role wins over the dealer tag" as the documented contract.

**Codex false-positive checks (confirmed correct as-shipped):**
- `admin:access` fall-through to `return false` is semantically correct — admins always exit at the shortcut at line 60; non-admins are denied.
- The resource-relative branch (`coach-availability:edit-own`) is still exercised by the existing pre-0036 test cases.
- TypeScript exhaustiveness still holds; adding a 23rd capability without a switch case would produce a compile error.
- Wiki preview is consistent with code (the `app:access` row's notes column documents the full `STAFF_APP_ROLES` admit set even though the table headers are admin/coach/dealer).

- **Hybrid-role coverage relies on the test fixtures matching production reality.** The schema permits any combination of roles; there is no DB constraint forbidding e.g. `['admin', 'dealer']`. The new hybrid tests cover the practical combinations the UI can produce. If the team_member_roles schema ever gains a stricter check, the hybrids may become unreachable in practice — but the pinning is still correct as documentation of the predicate's intent.
