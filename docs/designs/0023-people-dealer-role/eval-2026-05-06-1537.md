## Codex Adversarial Review

**Critical:** None.

**High:**

1. **Re-grant-after-archive would leave the role archived.** `src/features/schedule/actions.ts:235`. With `onConflictDoNothing` and the unconditional `(contact_id, role)` unique index in `src/lib/db/schema/team-member-roles.ts:26`, a contact whose `dealer` role was previously archived (e.g., via `archivePerson`'s tx that archives every active `team_member_roles` row) would have `updateDealer`'s defensive insert silently skipped on the conflict — leaving them with an active dealer-side staff link but an archived role. **Status: fixed in-eval.** Replaced with `onConflictDoUpdate({ target: [contactId, role], set: { archivedAt: null, updatedById: userId } })` — un-archives on conflict and updates the actor. Tests + tsc still green.

**Medium:**

2. **Dealer-role grants from `createDealer` / `updateDealer` aren't audited.** `people/actions.ts` emits `user.role_changed` for role transitions; the schedule-actions parallel paths don't. **Status: deferred.** Wiring a `recordAudit` emit into both call sites needs to know whether the upsert actually changed state (granted-fresh vs un-archived vs no-op), which Drizzle's `RETURNING` shape doesn't expose cleanly without an extra SELECT. Adding a helper that does the SELECT-then-conditional-write + audit emit is a small refactor warranting its own pass. The Phase 2 backfill (commit `c8c586e`) also didn't emit audits for the 23 inserts. Treating this consistently means a follow-up that wires dealer audits across all paths (createDealer, updateDealer, the backfill script's documentation), not piecemeal in Phase 4. Carry-forward.

**Low:**

3. **Comment said "partial unique index" but the index is unconditional.** **Status: fixed in-eval.** Comment now says "the unconditional `(contact_id, role)` unique index" with a pointer to the schema file.

**Explicitly cleared (not issues):**
- `target: [columnRefs]` syntax: Drizzle correctly translates to `ON CONFLICT (contact_id, role)` per `node_modules/drizzle-orm/pg-core/query-builders/insert.js:105`. If the index disappeared, Postgres would error rather than silently skip.
- Hybrid `dealer + admin/coach` in the existing-link branch is the documented Phase 3 behaviour.
- Transaction atomicity: `createDealer`'s tx wraps everything; a teammate-role insert failure rolls back the dealer + contact + dealer_contacts. Acceptable failure mode.
- `createDealer` no-contact branch correctly skips the role insert (no `contactId`).

## Cross-Reference

The High #1 fix interacts with 0019 Phase 4's `archivePerson`. After this fix, the lifecycle is:
1. `archivePerson` archives all `team_member_roles` rows for a contact (including any `dealer` row).
2. Admin later adds the contact back via `updateDealer` → `onConflictDoUpdate` un-archives the dealer role.

That's the correct round-trip. Without the fix, step 2 would have silently skipped, leaving an active dealer-side contact with no active role — violating Phase 5's planned invariant.
