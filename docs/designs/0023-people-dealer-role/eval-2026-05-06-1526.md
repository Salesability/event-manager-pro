## Codex Adversarial Review

**Critical:** None.

**High:** None.

**Medium:**

1. **Missing `updatePerson` regression test for dealer-preservation when app access is off.** **Status: fixed in-eval.** Added `preserves dealer role when appAccess is off, drops only admin/coach` test to `actions.test.ts`. Verifies that a form submission of `roles=admin&roles=dealer` with no `appAccess` results in: (a) auth ban (admin role lost), (b) audit emit `user.role_changed { before: ['admin'], after: ['dealer'] }`, (c) no auth user creation. 147/147 tests now.

2. **Destructive-edit confirm copy inaccurate for dealer-still-ticked saves.** The confirm message said "Saving with no roles will end app access…" but fires whenever `wantsAppAccess = admin || coach` is false — including when dealer IS ticked (dealer doesn't grant app access). **Status: fixed in-eval.** Updated copy at `people-admin.tsx:329` from "Saving with no roles will end app access for X and ban their sign-in account" to "Saving will end app access for X and ban their sign-in account" — drops the inaccurate "with no roles" preamble. The confirm still fires on the same trigger; only the wording is corrected.

**Low:**

3. **Plan drift: `dealer + admin/coach` hybrid ships without the planned "are you sure?" confirm.** The plan's Open Question #1 envisioned treating dealer + (admin|coach) as a confirm rather than blocking. Phase 3 ships with no enforcement — admin + dealer is currently allowed silently. **Status: deferred.** Codex agreed it's acceptable for v1 because staff-app roles are explicitly distinguished from dealer at every authz layer (`STAFF_APP_ROLES` constant, `is_staff_member()` SQL whitelist). The hybrid case can surface naturally during the audit-log review and we'll resurface the question if it shows up in real data. Plan's open question retained for Phase 5/6 revisit.

**Explicitly cleared (not issues):**
- Wire-format restoration is correct: hidden inputs render only from checked state, no double emit, parser dedupes via `getAll('roles')` set conversion.
- Mutual-exclusion (Open Question #1) — current "allow combinations" is acceptable for v1 with the Low #3 note.
- Audit emit isn't suppressed: `updatePerson` snapshot read inside tx remains correct, `desiredRoles` uses the filtered role set, `rolesChanged` fires when before/after differ.
- Section-gate stale state: untick clears `dealerLinks` so re-tick shows empty; matches stated intent.
- Hidden role inputs + uncheck race: cancelling destructive confirm returns null before action runs, UI state untouched.
- Edit-mode default `dealer=true` for backfilled contacts (Phase 2 prerequisite): checkbox checked, hidden `roles=dealer`, Dealers section visible by default.
- Dealer-only contacts without app access don't hit the destructive confirm because `person.hasAppAccess` is false.

## Cross-Reference

Codex's regression-test concern (Medium #1) and the destructive-confirm copy (Medium #2) are independent — both were correctly identified and both fixed in-eval. The wire-format restoration (the precondition fix this phase exposed) is independently regression-proofed by the new `updatePerson` test.

The wire-format regression discovered during implementation (`4a4afbd refactor(forms)` silently dropped the `fd.append('roles', ...)` calls) is the more interesting finding from this phase — the form has been silently submitting `roles=[]` for some hours; nothing in the test suite caught it because the people-admin tests exercise `createPerson`/`updatePerson` directly with hand-crafted FormData rather than driving the form. Worth noting for the future: client-side smoke tests that actually submit the form would catch this kind of regression. Listed as a coverage caveat below.

## Coverage Caveats

- **No client-side form submission test.** The wire-format regression discovered during implementation (`4a4afbd refactor(forms)` dropped `fd.append('roles')` lines without adding hidden inputs) wasn't caught by the existing test suite because all people-admin tests construct FormData manually instead of driving the form. A Playwright/RTL test that actually submits the dialog would have caught this. Worth a follow-up to add at least one such test once the destructive-flow concerns from `web-test`'s read-only discipline can be sidestepped (e.g., create-then-cleanup fixture).
- **Edit-mode dealer default not browser-tested.** No `dealer` rows exist yet (Phase 2 backfill is downstream of Phase 3). Phase 6's smoke verification will exercise the edit-mode default path on a real backfilled contact.
