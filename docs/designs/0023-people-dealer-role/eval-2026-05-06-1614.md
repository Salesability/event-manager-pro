Existing tests that built on the "roles can be empty" assumption were updated:

- **Reframed:** `creates a contact with no app access (Sales-Coach-style flow)` → `creates a dealer-only contact with no app access`. Same scenario, but post-Phase-3 the way to create a contact-only person is to tick `dealer` (which doesn't imply app access), not to create a roleless contact.
- **Reframed:** `coerces roles=[] when appAccess is being toggled off (stale UI defence)` → `rejects stale-UI coach-without-appAccess (post-coercion empty roles)`. Pre-Phase-5 the action silently coerced + banned; post-Phase-5 the assertion rejects this combination outright.
- **Reframed:** `on→off app access bans the auth user and clears app_metadata.role` → `on→off app access bans the auth user when role transitions admin → dealer`. The destructive ban path now requires an explicit dealer-only role choice; no-roles-at-all is rejected by the assertion.
- **Added:** `createPerson rejects without any role`, `updatePerson rejects without any role` — explicit invariant tests.
- **Adjusted:** `rejects malformed dealer link`, `rejects an invalid dealer-contact role`, `treats trigger-won link as the same outcome`, `partial-success warning if auth.admin.createUser fails`, `rejects when contact is missing`, `rejects when contact is archived` — added a role to the FormData so the invariant assertion doesn't intercept the test's actual flow.

## Decision: app-level over DB trigger

Plan's working lean was DB trigger. Reasons to flip to app-level for the implementation:

| | DB trigger | App-level (chosen) |
|---|---|---|
| Coverage | Bulletproof — catches scripts, migrations, manual `INSERT`s | Only catches the call sites we wired |
| Migration ordering | Tightly coupled to backfill order | Independent |
| Test fixtures | Triggers complicate teardown | Plain code path |
| `archivePerson` carve-out | Needs trigger deferral or schema gymnastics | Trivial — assertion lives in `create*`/`update*`, not `archivePerson` |
| `adoptOrphanAuthUser` carve-out | Trigger would reject the legacy-recovery path | Trivial — no assertion on this action |
| Phase 2 backfill | Needs to run before trigger | Already ran independently (commit `c8c586e`); zero truly-orphans on dev |

Trigger remains a future option if a regression slips through the app layer. The wiki documents the invariant + carve-outs so any future contact-creating path knows to add the assertion.

## Cross-Reference

The Phase 5 invariant closes the round-trip with Phase 4's `onConflictDoUpdate` upsert: the upsert un-archives a previously-archived `dealer` row, and the assertion ensures every new/updated contact carries at least one role. Together they handle:

- Contact archived (all roles archived) → admin re-links via `updateDealer` → dealer role un-archived → invariant satisfied.
- Contact never had roles (legacy or orphan-adoption) → admin opens via `/admin/people` → form-guard requires ≥1 role → action-assertion as backstop.
