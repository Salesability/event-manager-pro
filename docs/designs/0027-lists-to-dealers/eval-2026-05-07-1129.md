## Codex Adversarial Review

Verbatim from `codex:codex-rescue`:

**Critical / High / Medium:** None.

**Low**

> Required-field messages are submit/change driven, not true blur driven. Locations: `src/features/dealers/dealer-form.tsx:71`, `src/features/people/people-admin.tsx:465`, `node_modules/@radix-ui/react-form/dist/index.mjs:226`. Evidence: Radix updates built-in validity on native `change` and `invalid` only; `Form.Message` renders from `validity?.[match]`. There is no blur handler in `Form.Control`. A blank required field that is focused and blurred without a value change will not emit `change` or `invalid`, so the `valueMissing` message stays hidden until submit. Impact: misses the stated UX spec for "after blur of an invalid field," though it does not affect submission or FormData. Fix sketch: either accept submit/commit-time validation, or add a local blur/touched layer for required fields and render the same message after blur.

**Resolution:** Accepting submit-time validation as the Phase 4 UX. Adding a blur/touched layer would require maintaining per-field touched state outside Radix Form, which defeats the purpose of "use Radix Form's stock pattern." The current behavior — user submits, fields with `valueMissing` get inline errors, errors clear on input — is the standard Radix Form story and is consistent across both forms. If a future UX review wants blur-driven errors, that's a separate chunk that can ship a `useTouched()` hook for both forms (and any future BookingForm).

**Confirmed Non-Issues (Codex source-trace evidence)**

1. **Server Action compatibility — verified at the source.** Both forms pass `action={formAction}` to `Form.Root` (`dealer-form.tsx:55`, `people-admin.tsx:429`). Radix spreads `rootProps` onto `Primitive.form` and its submit handler does NOT call `preventDefault` on the valid path (`node_modules/@radix-ui/react-form/dist/index.mjs:106,115`). React DOM then invokes function actions when the submit event isn't already prevented (`node_modules/react-dom/cjs/react-dom-client.development.js:19084`). Radix does not call `requestSubmit()` because it does not intercept the valid submit path.
2. **Field name propagation:** `Form.Control` derives `name` from `controlProps.name || fieldContext.name` (`react-form/dist/index.mjs:178,268`). With `asChild`, Radix Slot merges those props into the child input — confirmed via the Radix Slot source.
3. **`useActionState` + `Form.Submit`:** `Form.Submit` is just a `Primitive.button` with submit type (`react-form/dist/index.mjs:370`); `asChild` preserves the child button's `type`, `disabled`, and classes via Slot. PersonForm's `disabled={pending || !hasAnyRole}` and DealerForm's `disabled={pending}` both pass through.
4. **Custom controls inside `Form.Root`:** Radix Form Root only wires form-level `invalid`, `submit`, and `reset` handlers; it does NOT require every descendant control to be a `Form.Field` (`react-form/dist/index.mjs:110`). Radix Checkbox serializes its own hidden checkbox input when `name`/`value` are passed (`react-checkbox/dist/index.mjs:232`). Roles fieldset wire-format intact.
5. **Dialog interaction:** no Radix Form/Radix Dialog focus conflict found. Form's only focus calls are invalid/server-invalid paths (`react-form/dist/index.mjs:110,248`); Dialog close-focus restoration remains in the wrapper (`src/components/ui/dialog.tsx:101`); first inputs still carry `autoFocus`.
6. **Email `typeMismatch`:** Radix reads native `control.validity` and matches `typeMismatch` from that state (`react-form/dist/index.mjs:184,314`); empty email does NOT trigger typeMismatch (no custom empty-email rule).
7. **Cancel buttons:** `Dialog.Close` renders `type="button"` and closes on click without submitting either form.
8. **Dynamic `dealerLinks`:** Multiple hidden `dealerLinks` inputs remain native form controls; Radix does not rewrite submit FormData; React DOM submits `new FormData(form)` for the action path. No deduplication.
9. **Forward compatibility:** `@radix-ui/react-form@^0.1.8` is pre-1.0; review-gate any future upgrade. No current break.

## Post-eval addendum (2026-05-07 11:35) — blur-driven required messages

Per user decision (option (b)), added a `useTouched` hook inline in both forms that drives a custom blur-time message:

- `onBlur` of an empty required field → marks `touched[name] = true`
- `onChange` to a non-empty value → marks `touched[name] = false` (clears the warning)
- `onInvalid` (browser fires this on submit when native validation fails) → marks `touched[name] = true`, covering never-focused required fields

Replaced `<Form.Message match="valueMissing">` with a touched-driven `<span>` for `name` (DealerForm), `firstName` + `lastName` (PersonForm). Email's `<Form.Message match="typeMismatch">` stays — browser-native `validity.typeMismatch` drives correctly via Radix's stock match path.

**Re-verification:**

- `pnpm tsc --noEmit` — exit 0
- `pnpm lint` — 0 errors, 4 pre-existing warnings
- `pnpm test --run` — 150/150 in 2.52s
- Browser smoke: opened `/dealerships` Add dialog → focused on autofocused Dealership name → clicked into Contact first (blur) → "Dealership name is required." rendered inline. Typed `X` into Dealership name → message cleared. (Accessibility-tree evidence captured in conversation log.)

The hook is duplicated across the two forms (~25 lines each). Per CLAUDE.md "no premature abstraction" guidance, leaving it inline; if a third form (BookingForm) lands, factor to `src/lib/forms/use-touched.ts`. Codex's original Low explicitly named "the working pattern is fine if blur-driven isn't required" — user requested blur-driven, so the fix is in.
