## Codex Adversarial Review

Codex review of the last 5 commits on `main` (`1c22b66 → c62bd52`, ~2500 lines). Verbatim:

### Critical
None.

### High

**[High] Public coach-share schedules are enumerable**
- File: `src/app/share/coach/[id]/page.tsx:15`
- Bug: public route trusts sequential DB IDs:
  ```
  const coachId = Number(id);
  const coachCampaigns = campaigns.filter((c) => c.coachId === coachId && c.status !== 'cancelled')
  ```
  This route is explicitly public in `src/lib/supabase/middleware.ts:4`: `'/share/coach'`.
- Trigger: unauthenticated user requests `/share/coach/1`, `/share/coach/2`, etc. and sees coach names plus assigned dealer schedule.
- Fix sketch: use an unguessable public share token/publicId, not `contacts.id`.

### Medium

**[Medium] Dev email delivery fails open to real recipients**
- File: `src/lib/email/send.ts:29`
- Bug: redirect only happens when the opt-in flag is set:
  ```
  if (process.env.APP_ENV === 'production') return false;
  process.env.EMAIL_FORCE_DEV_REDIRECT === 'true' && !!process.env.EMAIL_DEV_TO
  ```
  Otherwise `resend.emails.send({ from, to, subject, ... })` uses the real `to`.
- Trigger: local/staging has `RESEND_API_KEY` and `RESEND_FROM_EMAIL`, but forgets `EMAIL_FORCE_DEV_REDIRECT=true`; Email Client sends to the real campaign contact.
- Fix sketch: fail closed outside `APP_ENV=production`, or require an explicit `EMAIL_ALLOW_REAL_SEND=true`.

> Note: this is a deliberate change in commit `1c22b66`. Codex argues the new opt-in-redirect default is the wrong default. Worth a re-think now while the sandbox sender constrains real damage.

**[Medium] Host header can forge emailed share URLs**
- File: `src/features/email/actions.ts:21`
- Bug: share URL origin comes from request headers:
  ```
  const host = headerList.get('host') ?? 'localhost:3000';
  return `${proto}://${host}`;
  const shareUrl = `${await siteUrl()}/share/coach/${coach.id}`;
  ```
- Trigger: signed-in attacker invokes the Server Action with `Host: attacker.example`; coach receives a legitimate email containing `https://attacker.example/share/coach/<id>`.
- Fix sketch: build email links from a configured canonical `APP_BASE_URL`, not request headers.

**[Medium] Cancelled campaigns can still be edited**
- File: `src/features/schedule/actions.ts:430`
- Bug: update ignores campaign state:
  ```
  .update(campaigns)
  .set({ ...input, updatedById: userId })
  .where(eq(campaigns.id, id))
  ```
- Trigger: open edit form, cancel the campaign from another tab/dialog, then submit the stale edit; cancelled historical record is mutated.
- Fix sketch: add a status predicate/version check to `updateCampaign`, e.g. only update `draft/booked`.

### Low

**[Low] Email sends are not idempotent**
- File: `src/features/email/actions.ts:71` (also `:107`, `:140`)
- Bug: actions call Resend directly with no idempotency record:
  `const result = await sendEmail({ to, subject, text, replyTo });`
- Trigger: double-click, retry, or replay the Server Action; recipient gets duplicate confirmations/share links.
- Fix sketch: persist an email-send request key per campaign/coach/template and suppress duplicates. (The 5.5 plan already lists `messages_sent` audit table as deferred — wiring this to also de-dup would close both gaps.)

**[Low] Month-boundary events can disappear from calendar ribbons**
- File: `src/app/(app)/calendar/calendar-view.tsx:211`
- Bug: ribbon render requires both endpoints to exist in the current 42-cell grid:
  ```
  const si = cellIndexMap[ev.startDate];
  const ei = cellIndexMap[ev.endDate];
  if (si === undefined || ei === undefined) continue;
  ```
- Trigger: campaign runs `2026-01-31` to `2026-02-02`; February 2026 grid starts on Feb 1, so `startDate` is missing and no ribbon renders for Feb 1-2.
- Fix sketch: render any overlapping event by clamping start/end to the visible grid range.

> This is the same bug already parked as the slot-pack clamp follow-up: [`docs/designs/closed/0015-calendar-grid-clamp/plan.md`](../closed/0015-calendar-grid-clamp/plan.md). Independent confirmation.

### Explicitly not an issue

- Email Server Actions are session-gated before lookup/send (`src/features/email/actions.ts`).
- Reply-to is not form-controlled; it comes from Supabase `getUser().email`.
- Templates are plain text, not HTML, so HTML template injection is not present.
- Resend API key is not logged or returned; only missing-env names and provider error messages.
- Admin pages are under authenticated `(app)` layout; lookup/availability mutations call `requireUserId()`.
- Only route handler is the Supabase auth callback; no new internal UI mutation route handlers.
- Raw Drizzle SQL fragments are static schema checks/index predicates, not user-input interpolation.
- No new `onConflict`/upsert path in lookup or availability code.
