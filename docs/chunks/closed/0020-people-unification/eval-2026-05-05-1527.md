## Codex Adversarial Review (first pass — pre-fix)

### Blockers (BOTH FIXED in working tree)

**1. Archive state not representable for contact-only people** — `src/features/people/people-admin.tsx:44`. The old `isActive()` only checked `authUser?.bannedUntil`, so a contact-only person archived via `archivePerson` (which intentionally leaves `contacts.archived_at` null per `lifecycle.md`) would show `active` forever and keep the Archive button.

**Fix:** replaced with a three-state `lifecycle()` derivation: `active` if any facet is live (auth+not-banned, OR roles, OR dealer links), `banned` if the auth user is banned, `inactive` if no facets. Status chip + Archive button visibility both drive off this. Brand-new contact-only people with no facets show `inactive` from the start, which is honest — they are.

**2. Partial-success errors leave UI stale and contradict the toast** — `src/features/people/actions.ts:343, :352, :484`. `createPerson` and `updatePerson` could commit DB changes, then return `{ error }` when auth-side provisioning failed. UI did `toast.error` only — no refresh, no close — so messages like "Person created… open the row and retry" pointed to a row the admin couldn't see.

**Fix:** `ActionResult` is now `{ ok: true, contactId, warning?: string } | { error: string }`. The four partial-success paths (`auth.admin.createUser` fail, `updateUserById` ban fail, `syncAuthMetadata` fail in two branches, archive-side ban fail) return `{ ok: true, warning }` so the UI closes + refreshes + surfaces the warning. The race-loss compensation keeps `{ error }` (the operation didn't succeed) but now revalidates before returning so the next refresh shows the post-compensation state. UI's error branch also calls `router.refresh()` so the table updates regardless.

### Should Fix Before Phase 4 (carried forward)

**3. App-access toggled off traps checked roles in disabled state** — `people-admin.tsx`. If an existing admin/coach unchecks App access, the role checkboxes become disabled but stay checked in React state; submit fails with "clear the role checkboxes," which the user can't do.

*Status:* not yet fixed. Phase 2's server-side coercion (`roles=[]` when `appAccess=false` in `updatePerson`) means the action still does the right thing, but the UI is misleading. Two-line fix: `useEffect` watching `appAccess` to clear `admin` + `coach` when it flips false.

**4. App-access disable has no confirm step** — `people-admin.tsx`. Same edit form for routine name/email changes can submit App access off, which bans the auth user (`ban_duration='876000h'`). Should add a `confirm()` when `person.hasAppAccess && !appAccess` at submit.

**5. Edit form reconstructs authoritative names from generated `displayName`** — `people-admin.tsx`. `displayName` is a generated column (`first_name || ' ' || last_name`). The form splits it by whitespace, so a contact whose first name has spaces gets rewritten on save. Fix: expose `firstName` + `lastName` on `AdminPersonRow` and prefill from those raw fields.

**6. Server/client error message alignment** — minor copy work after #3 lands.

### Nit

**Test fidelity** (carry-forward) — the predicate-blind Drizzle mock means tests can't observe predicate regressions. Same shape across 0018 + 0020.

## Cross-Reference

The two blockers were related — both stemmed from the contract gap between "what the action did" and "what the UI shows next." The fixes are coordinated: the `warning` variant lets the UI close+refresh on partial success (so the admin sees the new row), and the `lifecycle()` helper lets the UI represent post-archive state correctly even when the DB row is technically still "live." Both shipped in the same fix cycle.
