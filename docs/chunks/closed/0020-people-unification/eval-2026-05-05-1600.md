## Codex Adversarial Review (first pass — pre-fix)

### Mediums (BOTH FIXED in working tree)

**1. `adoptOrphanAuthUser` raw DB constraint errors** — `src/features/people/actions.ts:645-679`. The action only validated `userId` non-empty, then inserted into `contacts.user_id` (FK to `auth.users`). Invalid UUIDs would hit Postgres `22P02 invalid_text_representation`; missing auth.users → `23503 foreign_key_violation`; concurrent adoption races into `contacts_user_id_unique` (code `23505`). The `toActionResult` catch only handled `IdentifierConflictError` and rethrew everything else.

**Fix:** added `UUID_RE` regex check before any DB call (`if (!userId || !UUID_RE.test(userId)) return { error: 'Invalid auth user id.' }`). Extended `toActionResult` to map `22P02` / `23503` / `23505` to admin-readable toasts:
- `22P02` → "Invalid id format."
- `23503` → "Referenced row no longer exists. Refresh the page."
- `23505` → "Another writer just made the same change. Refresh and check the row."

This also benefits `createPerson`, `updatePerson`, `archivePerson` since they share `toActionResult`.

**2. `scripts/adopt-orphan-auth-users.ts --auto` not atomic** — the per-orphan adoption did contacts insert + identifier insert as two separate statements. If the identifier insert hit the global active-uniqueness index (someone else already had that email as a primary), the contact stayed linked, and future dry-runs would silently filter it out as "already linked."

**Fix:** wrapped the per-user adoption in `db.transaction(...)` so an identifier conflict rolls back the contact too. Plus per-orphan try/catch + summary log (`{n} adopted, {m} failed`) so one failure doesn't abort the batch.

### Lows (deferred to Phase 5)

**3. Wiki/code drift** — `docs/wiki/auth.md` still describes `/admin/users` and "Link contact." Phase 5's checklist explicitly schedules the rewrite, so this is doc drift, not a Phase 4 bug.

**4. `loadOrphanAuthUsers` 1000-user pagination cap** — same as `loadAdminPeople` and the legacy `loadAdminUsers`. Acceptable at current scale; revisit if the project ever exceeds.

### Nit (deferred)

**5. Stale `revalidatePath('/admin/users')` in `revalidatePeopleViews()`** — harmless (the path now redirects), but noisy. Comment-only placeholders in `auth/actions.ts` and `schedule/actions.ts` ditto.

### Verified Non-Issues (Codex confirmed)

- No live imports of deleted `auth/queries`, `users-admin`, `coach-form`, or removed actions remain.
- `/admin/users` redirect chain is sound: middleware admin gate fires for non-admins → `/`; admins reach the redirect Server Component → `/admin/people` re-runs `requireAdmin`.
- `headerAddClass` and `rowDeleteClass` constants in `lists/list-actions.tsx` are still used by the dealer row — not orphaned.
