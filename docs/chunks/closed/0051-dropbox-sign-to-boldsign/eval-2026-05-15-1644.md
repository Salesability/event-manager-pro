## Codex Adversarial Review

### High

No High findings.

### Medium

1. **Signable-field placement absent (introduced by 0051).** `src/lib/boldsign/client.ts` `sendSignatureRequest` sets title/message/sandbox/signer/files/metadata only — never `formFields`, `useTextTags`, `autoDetectFields`, or `enableAllowSignEverywhere`. The rendered MSA at `src/lib/pdf/render-msa.ts:296` contains only visual signature lines, not provider tags. BoldSign's SDK defaults `useTextTags`/`autoDetectFields` to false. Hypothesized impact: BoldSign may reject the send or produce an envelope the signer cannot complete; the signed webhook path then never fires. The unit mock at `src/lib/boldsign/client.test.ts:16` cannot catch this because it accepts any `sendForSign` object and returns `documentId` unconditionally. **Verification requires live sandbox round-trip.**

### Low

1. **Non-ASCII `s0` may throw inside `timingSafeEqual` (introduced by 0051).** `src/lib/boldsign/webhook-verify.ts:52` `constantTimeEquals` checks JavaScript string length before calling `timingSafeEqual(Buffer.from(..., 'utf8'))`. A 64-character non-ASCII signature candidate has the same string length as the hex digest but a longer UTF-8 byte length, so `timingSafeEqual` can throw instead of returning the 401 verification error. No signature bypass observed; this is malformed-input handling / availability.

2. **Plan-of-record grep-gate hygiene (introduced by 0051).** Plan's Phase 7 final-verification claim that `rg -rin "dropbox" src/ .env.example` returns zero is contradicted by intentional matches in `.env.example:53` (chunk-slug reference) and `src/features/msa/template-version.ts:11` (historical comment). Plan-doc only — no functional impact.

### Clean areas (no findings observed)

- Drizzle rename / snapshot consistency — SQL is rename-only; schema uses the renamed nullable text column; 0020→0021 snapshot diff changes only snapshot IDs + the column key/name.
- Webhook raw-body ordering / caller-state boundary — route reads `request.text()`, verifies HMAC, then parses JSON; lifecycle dispatch uses only the verified payload's document id for DB lookup/update.
- `providerDocumentId` guarded UPDATE rename — `sendMsaEnvelope` still guards persist on `status='pending' AND providerDocumentId IS NULL`; lifecycle transitions key on the renamed column.
