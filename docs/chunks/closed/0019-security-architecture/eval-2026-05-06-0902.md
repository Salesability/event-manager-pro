## Codex Adversarial Review

**Critical:** None.

**High:** None.

**Medium:**

1. **Residual pre-existing policies would survive, including anon policies.** `drizzle/0003_enable_rls.sql:62,68`. The migration drops only the two expected policy names per table; if a dev/prod DB has any manual/earlier policy under a different name, enabling RLS would not give the intended default-deny baseline. Migration-safety gap. *Suggested fix: add a post-migration assertion that `pg_policies` contains exactly the 22 expected policies, or explicitly drop all pre-existing policies before recreating.*

2. **Integration test can pass while over-permissive policies are broken on an empty DB.** `tests/integration/rls.test.ts:80`. The proof is `count(*)::text FROM table` followed by `expect(...).toBe(0)`. If a table is empty, `USING (true)` for `authenticated` still returns 0, so denial is not actually proven for each table. Test also doesn't assert anon default-deny. *Suggested fix: insert a sentinel row per table inside the bypass-role transaction, then assert authenticated non-staff and anon cannot see it.*

**Low:**

3. **Drizzle BYPASSRLS invariant is only tested, not enforced at app startup.** `src/lib/db/index.ts:10`. A future `DATABASE_URL` change to a non-BYPASSRLS role would silently break staff code. *Suggested fix: add a startup/admin health assertion that `current_user`'s `rolbypassrls=true`.*

4. **Drizzle snapshot contradicts the migration's RLS state.** `drizzle/meta/0003_snapshot.json:264,271`. Snapshot shows `"policies": {}` and `"isRLSEnabled": false` for tables we just RLS-enabled — Drizzle can't model this. Doesn't affect runtime; can mislead future migration generation. *Suggested fix: regenerate snapshot or document the limitation.*

**Explicitly cleared (not issues):**
- search_path injection on `is_staff_member()` — pinned to `''`, schema-qualified everywhere.
- WITH CHECK / USING parity — all policies match.
- All 11 tables covered by both intended policies (22 CREATE POLICY total).
- `auth.uid()` semantics inside SECURITY DEFINER — confirmed connection-level GUC.
- `sql.begin` reserves a single connection; `SET LOCAL ROLE` does apply to subsequent `tx` calls.
- `/share/coach/[id]` queries via Drizzle, not via PostgREST anon — anon default-deny is fine.
- Anon-key Supabase clients in `src/lib/supabase/{server,client,middleware}.ts` are auth-only — no `.from(...)` or `.rpc(...)` against domain tables.
