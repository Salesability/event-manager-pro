## Codex Adversarial Review

**Critical:** None.

**High:**

1. **Audit-after-commit failure mode.** `archiveDealer`, `cancelCampaign`, `archivePerson`, `updatePerson` all call `recordAudit()` AFTER the mutation/auth work has committed. If `recordAudit` rejects, the action rejects without revalidation but the mutation persists â€” leaving the user with an error toast over a successfully-committed mutation. **Status: fixed in-eval.** `recordAudit` itself now wraps the DB insert in try/catch and logs failures via `console.error` rather than throwing. This deviates from plan Decision #3's "same transaction" framing; the deviation is documented in the helper's docstring and in eval. Rationale: the wired actions span DB + Supabase auth boundaries (createUser / ban / metadata), so wrapping audit in the mutation tx wouldn't actually keep the operation atomic. Best-effort with server-log gap reporting is the pragmatic fit.

**Medium:**

2. **`createPerson` initial role grants weren't audited.** **Status: fixed in-eval.** Added `recordAudit({ action: 'user.role_changed', payload: { before: [], after: roles } })` after the create-tx commits, when `roles.length > 0`. Fires even on partial-success warning paths (the role rows ARE committed; the auditable event is the role grant).

3. **`updatePerson` onâ†’off auth ban wasn't audited as `user.deactivated`.** **Status: fixed in-eval.** Added `recordAudit({ action: 'user.deactivated', payload: { authUserId, via: 'updatePerson' } })` after the ban succeeds, alongside the existing role-diff audit. Two emits when both apply (deactivation AND role change).

4. **Role-diff snapshot was outside the tx.** A concurrent `updatePerson` between the snapshot read and `syncTeamMemberRoles` could make `before` inaccurate. **Status: fixed in-eval.** Moved the snapshot SELECT INSIDE `db.transaction(...)`; closure-captured `existingRoles`/`desiredRoles`/`rolesChanged` for the post-tx audit emit. Test queue ordering preserved.

**Low:**

5. **`adoptOrphanAuthUser` not in audit list.** **Status: deferred.** Identity-binding event but not in Phase 4's scope. Noting as a follow-up; revisit when an admin UI for the audit log lands and a complete event taxonomy is defined.

6. **`actorRole` denorm copies only `app_metadata.role` â€” null for non-admin actors.** **Status: deferred.** Today every audited action is `requireRole('admin')`-gated, so non-admin actors don't reach `recordAudit`. The user_id FK is the primary key of "who did this" anyway; `actor_role` is supplementary. Worth tightening when non-admin paths gain audited actions.

7. **`target_id` is nullable.** **Status: deferred.** All wired emits pass non-null IDs; the schema allows null for forward-compat (bulk events). Could tighten to NOT NULL now and re-loosen later, but keeping nullable matches our "permissive schema, strict callers" posture for jsonb-shaped audit logs.

**Explicitly cleared (not issues):**
- `archiveDealer.returning()` doesn't change the action's contract â€” return shape is `{ ok: true }` either way; `.returning()` is consumed internally to suppress duplicate audits on no-op archives.
- RLS policy scoping is sound. `service_role` permits all (Postgres role gate, not impersonable from JWT). `authenticated` SELECT is `actor_user_id = auth.uid()`. Rows with `actor_user_id = NULL` (post-FK SET NULL) are correctly invisible to authenticated.
- `payload` jsonb uses Drizzle's parameterized insert path â†’ no JSONB injection.
- Drizzle journal/snapshot chain is intact: `0003.prevId â†’ 0002.id`, `0004.prevId â†’ 0003.id`, journal idx-ordering correct after the `when` bump fix.
