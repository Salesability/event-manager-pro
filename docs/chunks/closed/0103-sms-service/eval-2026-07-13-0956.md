## Codex Adversarial Review

(verbatim)

**Critical**

None.

**High**

1. `src/features/sms/actions.ts:151`: opt-outs can race the launch evaluation and still receive a text. `launchSmsSend` evaluates opt-out state once at lines 151-155, creates rows at 169-193, then sends from the stale `eligible` array at 201-213. A STOP webhook or manual opt-out inserted after evaluation via `route.ts:145` or `actions.ts:270` is not rechecked before `sendSms`. Smallest fix: before each Twilio call, re-check `sms_opt_outs` for that phone; for the strict "never after STOP" invariant, take a per-phone advisory lock in both send and opt-out paths, then re-check under the lock before dispatch.

2. `src/features/sms/actions.ts:169`: double-click / double-submit launches duplicate sends. Every invocation inserts a fresh `sms_sends` row and fresh `sms_messages` rows at 169-193; `sms-sends.ts:16` has only non-unique indexes, and the client-side pending button at `sms-panel.tsx:100` is not a server-side idempotency control. Two rapid action posts can text the same eligible list twice. Smallest fix: require a launch idempotency key with a unique DB constraint, or take a campaign-level launch lock / "active send" guard server-side before creating rows.

3. `src/features/sms/actions.ts:209`: crash after Twilio accept but before `providerSid` update loses the correlation and can cause duplicate follow-up sends. The external send happens at 209-213, then the SID is persisted at 216-219. If the process dies in that gap, Twilio may deliver the message, but the ledger row remains `queued` with no SID. A callback for that SID cannot match and returns 404 at `route.ts:120`, and staff may relaunch, texting that recipient again. Smallest fix: move dispatch into a resumable outbox/worker with explicit per-row attempt states and recovery, or use a provider-supported client reference/idempotency mechanism if available; do not rely on request lifetime for durable dispatch.

**Medium**

1. `src/features/sms/import-csv.ts:93`: future `last_contact_at` values bypass consent staleness. The parser only checks `YYYY-MM-DD` shape at 93-99, stores the date at 149-155, and `eligibility.ts:48` treats any implied-consent date whose close window is after `now` as eligible. A CSV typo like `2099-01-01` makes an old implied-inquiry recipient eligible for decades. Smallest fix: validate `last_contact_at` is a real calendar date and not after import/launch date; reject future dates for implied consent.

2. `src/features/sms/actions.ts:106`: Twilio's 1600-character cap is checked before appending the mandatory STOP footer. `launchSchema` allows a 1600-char body at 106-110, then `withStopFooter` appends text at 115-117, so an otherwise valid form can exceed the provider limit and fail every recipient at dispatch. Smallest fix: apply the footer before validating length, or reduce the user-entered max by the footer length when the footer will be appended.

**Low**

None.

**Examined Not Issues**

- `src/lib/auth/capabilities.ts:119`: `sms:send` is admin-only; non-admin roles reach the switch case and return false.
- `src/features/sms/actions.ts:57`: all three SMS mutations use `capabilityClient('sms:send')`; no ungated Server Action path found.
- `src/app/(app)/calendar/[id]/sms/page.tsx:24`: the SMS page enforces `assertCan('sms:send')`, and launch still re-checks campaign status/add-on server-side.
- `src/lib/sms/webhook-verify.ts:27`: locally matches Twilio's documented HMAC-SHA1 URL + sorted params scheme; tampered values and Host-confusion signatures are covered in `webhook-verify.test.ts:48`. Hypothesis label: this assumes Twilio sends no duplicate form keys that must be preserved.
- `src/app/api/twilio/webhook/route.ts:61`: webhook verifies signature before DB reads/writes.
- `src/app/api/twilio/webhook/route.ts:103`: out-of-order status callbacks are monotonic; delivered followed by sent remains delivered in `sms-service.test.ts:214`.
- `src/app/api/twilio/webhook/route.ts:131`: unknown SIDs return 404 as intended for raced callbacks.
- `src/lib/db/schema/sms-recipients.ts:57`: E.164 CHECK constraints use static Drizzle SQL fragments, not user-controlled raw SQL.

- `src/features/sms/import-csv.ts:39`: quoted commas and doubled quotes are handled for single-line CSV fields; malformed rows reject all-or-nothing before import.
