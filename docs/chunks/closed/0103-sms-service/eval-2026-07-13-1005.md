## Residual findings — accepted tradeoffs (not blocking)

- **Medium — the 60s guard is a window, not a launch-in-progress lock.** A duplicate launch fired >60s after the first (while a very long dispatch loop is still running) isn't blocked, and a legitimate back-to-back send within 60s is (with a clear error pointing at the send log). A true in-progress lock needs an "active send" status column (schema change) — folded into the parked **0103-a** durable-dispatch follow-up, which owns this whole reliability surface. At v1 list sizes (hundreds; dispatch well under 60s) the window covers the actual double-click failure mode.
- **Low — advisory-lock key uses `hashtext` (32-bit space).** A cross-campaign hash collision merely serializes two unrelated launches briefly; the 60s check itself is campaign-id-scoped, so no correctness impact. Not worth a two-key refactor.
