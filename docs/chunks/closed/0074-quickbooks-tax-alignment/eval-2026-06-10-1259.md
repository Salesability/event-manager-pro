## Codex Adversarial Review

(Verbatim severity; disposition added by the orchestrator.)

### High
1. **Wrong province auto-linked when multiple app provinces share a rate but only one QBO code exists** (`tax-sync.ts` `matchProvinceTaxCode`/`applyTaxCodeSync`). E.g. BC + MB are both 12%; with a single 12% QBO code, the per-province matcher (counting matching *codes*, not *provinces*) links both to that one code → wrong jurisdiction. The documented limitation only covered >1 *code* at a rate; this province-side collision mislinked instead of failing closed.
   - **Disposition: FIXED IN-CYCLE at `963ee2e`.** Added `resolveProvinceLinks` — a code claimed by >1 province is now `ambiguous` (unlinked) for all claimants; only a 1:1 province↔code rate match auto-links. `applyTaxCodeSync` uses it. 3 unit tests (incl. the BC+MB collision). Rate-collision provinces now require manual mapping (still a follow-up) but are never mislinked.

### Medium
1. **Sparse update can leave a stale tax code when a quote becomes untaxed** (`quote-push.ts` omits `TxnTaxDetail` when untaxed; `client.ts` `updateEstimate` always sends `sparse: true`). Re-pushing a now-untaxed quote (`tax='0'`) onto an Estimate that already has a `TxnTaxCodeRef` → the sparse merge can retain the old code, so QBO keeps taxing.
   - **Disposition: PARK (0074 follow-up a).** Rare transition (taxed→untaxed re-push); the create path + taxed-update path are correct. Reliably *clearing* QBO tax on a sparse update needs a live CA-sandbox round-trip to confirm the mechanism (same flavor as the 0073-a sparse-line question we settled by live testing). Fix direction: on an untaxed update, explicitly clear the Estimate's tax (e.g. full update, or the right TxnTaxDetail/GlobalTaxCalculation to zero it). **Un-park with the live CA smoke.**

### Low
1. **The drift guard passes one-cent mismatches** (`quoteTaxMatchesRate`, `≤ 0.01` tolerance). A $52.00 snapshot vs a $52.01 QBO computation passes.
   - **Disposition: INTENTIONAL — no change.** The cent tolerance absorbs legitimate rounding; tightening it would false-positive on normal rounding and block valid pushes. The arithmetic is otherwise correct (Codex confirmed `400×13→52.00`, `100×14.975→14.98`).

### Explicitly NOT an issue (Codex, confirmed)
- `resolveCodeRatePct` sums GST+PST groups correctly + returns null on an unresolvable `TaxRateRef`; QC 14.975 survives the `<0.001` tolerance; inactive codes filtered.
- Seeded provinces have no 0% rate + untaxed quotes omit `TxnTaxDetail`, so exempt/zero-rated codes don't affect normal pushes.
- `applyTaxCodeSync` clears stale links; the `loadQuoteEstimatePushData` province join can't fan out (province UNIQUE), null province → null fields handled by the taxed-quote pre-flight.
- `taxCodeId as string` cast is sound after the `taxed` guard; `pullTaxCodesFromQuickbooks` is admin-gated before side effects; the probe script's inline decrypt is acceptable for a throwaway read-only research script.
