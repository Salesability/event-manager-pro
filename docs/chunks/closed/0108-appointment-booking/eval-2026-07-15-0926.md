## Codex Adversarial Review

Verbatim findings:

**High**

- `src/features/sms/actions.ts:114` + `src/features/bookings/book.ts:89` + `src/lib/db/schema/appointments.ts:96`: a recipient re-import can bypass "one appointment per person." `importSmsRecipients` wholesale deletes the campaign's `sms_recipients`; the existing appointment survives with `recipient_id = NULL` via `ON DELETE SET NULL`, so the partial unique index no longer applies. Saving booking settings then mints a new token for the reinserted same phone/person, and `bookSlot` only checks for an existing appointment by the new `recipientId`. Failure scenario: Sarah books slot A, staff re-imports the same CSV, Sarah receives/copies the new token, and books slot B; both live appointments remain because the first row is now keyed only by the snapshotted phone/name.

**Medium**

- `src/app/(app)/calendar/[id]/bookings/page.tsx:20` + `src/features/bookings/actions.ts:122` + `src/features/bookings/book.ts:51`: booking enablement is only UI-gated for booked SMS-addon campaigns. The event dialog hides "Bookings" unless `campaign.status === 'booked' && smsEmail > 0`, but the bookings page itself only checks `assertCan('sms:send')`, and `saveCampaignBookingSettings` rejects only `cancelled`. Failure scenario: a staff user with `sms:send` deep-links to `/calendar/<draft-or-no-sms-addon-id>/bookings`, enables settings, gets recipient tokens minted, and customers can book appointments on a campaign that the SMS surface would refuse.

- `src/features/schedule/actions.ts:962` + `src/features/bookings/queries.ts:55` + `src/features/bookings/book.ts:51`: existing tokens continue to resolve and book after a campaign is cancelled. Cancellation only updates `campaigns.status`; `loadBookingContext` and `bookSlot` do not select or check campaign status, only `endDate`. Failure scenario: staff enables bookings for a booked event, later cancels it, and a customer uses the old `/book/<token>` link before the event end date; the page still shows slots and `bookSlot` inserts a live appointment for the cancelled event.

- `src/features/bookings/book.ts:51` + `src/features/bookings/book.ts:101` + `src/features/bookings/actions.ts:133`: lowering capacity can race with an in-flight booking because settings saves use the `sms_launch_` lock, while bookings use the `booking_` lock and carry a pre-transaction `slotCapacity` snapshot into the locked section. Failure scenario: capacity is 2 with 1 booked; customer submit reads capacity 2; staff lowers capacity to 1 and commits; customer's transaction then locks `booking_`, counts 1, compares against stale capacity 2, and inserts a second appointment, leaving the slot at 2/1.

**Low**

- No low-severity findings observed.

**Explicitly Not An Issue**

- `src/lib/supabase/middleware.ts:22`: the `/book` public prefix is broad, but the current route inventory only contains `src/app/book/[token]/page.tsx`; no existing sensitive `/book/*` route exposed.
- `src/features/bookings/actions.ts:59`: token is `encodeURIComponent`-wrapped before redirecting back to `/book/<token>` — no open redirect.
- `src/features/bookings/book.ts:35` / `actions.ts:136`: advisory lock SQL uses Drizzle parameter interpolation — no injection in the lock key.
- `src/app/book/[token]/page.tsx:30` + `queries.ts:76`: the public page resolves exactly one token row — no other-recipient PII leakage.
