## Codex Adversarial Review

**Critical:** None.

**High:**

1. **`dealer` aliased to staff in `is_staff_member()` and the staff-app gate.** Adding `dealer` to the `team_member_role` enum + `V1_TEAM_ROLES` without filtering it out of the staff-app role set meant a dealer-only `team_member_roles` row would have passed:
    - `requireStaffAccess()` (page gate at `(app)/layout.tsx`),
    - `auth/callback/route.ts` role check (would land them on `/calendar`),
    - `is_staff_member()` in `drizzle/0003_enable_rls.sql` (the `<table>_staff_all` RLS policy returns all rows for any active `team_member_roles` row).

   **Status: fixed in-eval.** Three coordinated changes:
    - New `STAFF_APP_ROLES = ['admin','staff','coach','viewer']` constant + `isStaffAppRole()` predicate exported from `src/lib/auth/load-team-membership.ts`.
    - `requireStaffAccess()` now checks `membership.roles.some(isStaffAppRole)` (was `roles.length > 0`).
    - `src/app/auth/callback/route.ts` mirrors the same filter.
    - New `drizzle/0006_is_staff_member_excludes_dealer.sql` rewrites the SQL helper as `CREATE OR REPLACE FUNCTION` with `tmr.role IN ('admin','staff','coach','viewer')`. Whitelist (default-deny new enum values) rather than blacklist.
    - Applied to dev DB; `pg_get_functiondef` confirms the new body. New vitest case (`route.test.ts:dealer-only → does NOT land on the staff app (0023 regression guard)`) asserts the gate.

**Medium:**

2. **Backfill rows can be archived by stale UI between Phase 2 and Phase 3.** Once `dealer` is in `V1_TEAM_ROLES`, the archive filter in `syncTeamMemberRoles` includes it. If Phase 2's backfill creates dealer rows before Phase 3 adds the form checkbox, any `updatePerson` call (with the Phase-2-but-pre-Phase-3 form) submits `roles=['admin'|'coach']` (no dealer) → the freshly-backfilled dealer row gets archived. **Status: documented in the plan's Phase 1 checklist.** Mitigation: ship Phase 3 (UI) before Phase 2 (backfill), OR ship 1+2+3 as one deploy. Worth flagging in the Phase 2 plan text too as a hard prerequisite.

3. **Hand-maintained `TeamMemberRole` union drift.** `src/lib/auth/load-team-membership.ts:8` duplicates the enum literals manually, separate from `teamMemberRole.enumValues` in the schema. Easy to forget on the next enum change. **Status: deferred (Codex Medium #4).** Worth deriving from `(typeof teamMemberRole.enumValues)[number]` in a follow-up; not load-bearing for Phase 1's correctness since both surfaces now agree on the 5-value set.

**Low:**

4. **Migration missing `IF NOT EXISTS`.** The plan note claimed the `ADD VALUE` had `IF NOT EXISTS` for idempotency, but the file actually has the bare `ADD VALUE 'dealer'`. **Status: corrected in the plan note (in-eval).** Not load-bearing — drizzle-kit's migration runner records hashes and won't re-apply, and re-running against another DB where `dealer` already exists would error loudly rather than silently misbehave (acceptable for the team's actual workflow). Worth tightening on the next `ALTER TYPE ADD VALUE` migration.

5. **`requireRole('dealer')` future footgun.** Widening `TeamMemberRole` to include `dealer` makes `requireRole('dealer')` compile. No call site uses it today, but a future maintainer could accidentally let dealer-only users into a staff Server Action. **Status: deferred (Codex Low #5).** Worth introducing a narrower `StaffActionRole = Exclude<TeamMemberRole, 'dealer'>` in a follow-up. Practical risk is low while the staff-app gates filter dealer out (which they now do).

**Discovered (not Codex-flagged in this run, but called out by Codex's prior 0023-scaffold review):**

6. **Wiki drift.** `docs/wiki/data-model.md:11`, `:24`, `:132` and `docs/wiki/auth.md:75`, `:79` still describe the role taxonomy as 4 values. **Status: deferred to Phase 6 of the plan.** Phase 6's smoke verification + the `docs/wiki/auth.md` rewrite (Phase 8 of 0019 set the precedent) should land both updates together once the chunk is shipping.
