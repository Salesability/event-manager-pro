## Codex Adversarial Review

Codex (codex:codex-rescue, read-only) reviewed the full chunk diff. The final severity synthesis truncated on flush (known codex-rescue behavior — reconstructed from the job log). Findings:

**Medium — decision flags trusted without proof of a prior match.** `reuseContactId` and `linkQuickbooksId` are accepted by `createDealer` as authority rather than as proof the action actually surfaced that match. A forged hidden field could link an arbitrary contact / born-link an arbitrary QB Customer the UI never offered (Codex note: `archivePerson` archives relationships not the contact row, so a forged reuse could re-add a dealer relationship for a previously-archived person). Codex explicitly **declined to frame this as privilege escalation** — the paths are admin-only (`dealer:create`/`dealer:edit`) in a single-tenant app, and an admin can already create such links through legitimate UI; it's a data-integrity hardening gap.
- **`reuseContactId` half — FIXED in-cycle (`eba6126`):** the action now re-runs `findExistingContactByIdentifier` and drops the flag unless it equals the contact that holds the submitted email/phone, re-surfacing the real duplicate instead. (+ a forge-rejection unit test.)
- **`linkQuickbooksId` half — PARKED (0085-a):** validating an arbitrary QB Customer Id needs a QB read (`fetchCustomerById` + DisplayName compare) that conflicts with the best-effort principle (a QB outage shouldn't block a legit link). Deferred.

**Non-issues (Codex false-positive reduction):**
- **QBO query injection — NOT an issue.** Codex verified the backslash-escaping in `findCustomerByDisplayName` (`\\` then `'`): "a quote preceded by any number of user backslashes becomes an odd number of backslashes before the quote after escaping, so the usual breakout payload does not terminate the literal if QBO honors backslash escapes."
- **Drizzle SQL fragments — parameterized.** The `lower(trim(...))` fragments in `dedup.ts` bind values; no raw user-input interpolation.
