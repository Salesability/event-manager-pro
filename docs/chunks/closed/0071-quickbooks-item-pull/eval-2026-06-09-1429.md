## Codex Adversarial Review

(Verbatim severity + finding; disposition added by the orchestrator.)

### High

1. **Non-empty partial QBO reads can still destructively sync** (`client.ts` `fetchItems` + `item-sync.ts` `applyItemSync`). The `items.length === 0` guard didn't fire when a transient HTTP-200 short/empty page (or a response of only non-syncable items) produced a non-empty-but-incomplete set, so the apply would archive the "missing" linked rows + purge the catalog.
   - **Disposition: PARTIALLY FIXED IN-CYCLE at `7f86715` + residual PARKED (follow-up a).** The guard now aborts (no writes) when there are **zero *syncable* items** ŌĆö covering the empty-pull, all-non-syncable, and transient-empty-first-page cases (+ an integration test). The **residual** ŌĆö a *partial multi-page* read where page 1 is full but a later page silently returns empty (some syncable items present, others missing) ŌĆö is **parked**: it needs `fetchItems` pagination validation (detect an unexpectedly-short page) and/or a "would archive/purge most of the catalog ŌåÆ require confirm" threshold. **Un-park trigger: before enabling a prod item pull** (sandbox-only today; the pull is admin-initiated and not prod-deployed).

### Medium

2. **`applyItemSync` was unsafe when called outside a transaction** ŌĆö the purge-then-create body relies on caller atomicity; a future `applyItemSync(items)` on the bare pool could half-purge then fail.
   - **Disposition: FIXED IN-CYCLE at `7f86715`.** Removed the `= db` default so the executor is required (the Server Action passes a `db.transaction` handle; the integration test passes a rolled-back tx); doc-comment states the transaction requirement.

### Low

3. **QBO prices with >2 decimals churn (perpetual `update`) or out-of-range values abort the tx** ŌĆö `String(item.UnitPrice)` vs `numeric(10,2)`.
   - **Disposition: FIXED IN-CYCLE at `7f86715`** (churn). `mapItemToServiceItem` now canonicalizes via `UnitPrice.toFixed(2)`, so a `99.999` stores as `100.00` and the next pull compares equal (`current`, no churn). The magnitude-overflow (a service price > `99,999,999.99`) remains theoretically possible but is not a realistic catalog value ŌĆö accepted, not parked.

### Explicitly NOT an issue (Codex false-positive reduction, confirmed)
- Production action is transaction-wrapped (`actions.ts` `db.transaction((tx) => applyItemSync(items, tx))`) ŌĆö no mid-apply leak via that path.
- Partial-unique-index shape matches the `onConflictDoNothing` target (`quickbooks_id WHERE ŌĆ” IS NOT NULL`).
- `code` UNIQUE collisions guarded at classify time (linked-code + duplicate-create); legacy unlinked codes are freed by the purge-before-create ordering.
- Historical quotes protected: `quote_line_items.service_item_id` is `ON DELETE SET NULL` + snapshotted; **no other FK** references `service_items`.
- CRUD removal complete in `src` (no dangling `ServicesAdmin`/service-action refs); gate-matrix drift grep consistent (new pull-action row present, 3 item-CRUD rows removed).
- Auth gate precedes side effects (`assertCan('admin:access')` before token/fetch/transaction/redirect).
