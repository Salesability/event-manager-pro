## Codex Adversarial Review

**Blocker / High:** none.

**Concern (1, fixed in-eval):**

- Original implementation used `firstAdminIndex = tabs.findIndex(t => t.admin)` + `showSeparator = i === firstAdminIndex && i > 0`. Codex flagged that this assumed admin tabs are contiguous; if a future non-admin tab were inserted between two admin tabs, the divider would render only before the FIRST admin tab and the inserted non-admin tab would silently land in the visual "admin group". Resolution: changed the trigger to a per-iteration `tab.admin && prev && !prev.admin`, which renders a divider on every non-admin → admin transition. Handles arbitrary tab ordering without re-thinking the predicate; closes the future-defense hole and the comment now reflects the broader semantics.

**Nit (2, accepted as-is):**

- "Six-tab list" is the human shorthand in plan/eval prose, not a claim about HTML list semantics. The component is `<nav>` containing `<Link>` elements; screen readers announce navigation links, not list items. Plan wording softened to "six links" in next pass.
- `aria-hidden` on the divider is the correct call for a visual-only separator in a horizontal nav. `role="separator"` would add semantic noise without making the link list clearer to assistive tech. Codex agrees; no change.

### Codex "Not An Issue" (false-positive reduction)

- Wrapper `<span className="flex items-center gap-1">` around each link doesn't change the focus target — the link is the only focusable child, so Tab still lands on the link directly.
- A11y tree confirmed clean: `@e6 link "Calendar"` … `@e11 link "People"` in expected order; no extra refs from the wrappers or divider.
