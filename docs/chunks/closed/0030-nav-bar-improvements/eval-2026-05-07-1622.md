## Codex Adversarial Review

**Critical / High:** none.

**Medium (1, fixed in-eval):**

1. **Form-in-menu-item submission race (FIXED).** Original implementation wrapped the menu item as `<form action={signOut}><DropdownMenu.Item asChild><button type="submit">Sign out</button></DropdownMenu.Item></form>`. Codex flagged that Radix's onSelect handler closes/unmounts the portal on activation, which can race with the native form submit dispatch — particularly fragile under Enter/Space activation. Resolution: replaced with the canonical Radix-form pattern — keep the form wrapper, but make the menu item a regular `DropdownMenu.Item` whose `onSelect` calls `e.preventDefault()` (cancels Radix's default close) then `target.closest('form')?.requestSubmit()` (submits the parent form deterministically). Submission completes before unmount; the redirect from the Server Action handles the post-submit page transition naturally.

**Low (2, both fixed in-eval):**

1. **Long-email overflow (FIXED).** Email render only had `min-w-[14rem]` with no upper bound; a long staff email could push the menu portal off-viewport. Resolution: added `max-w-[20rem]` + `break-all`; existing `title={email}` retained as the full-text fallback.
2. **z-index conflict potential (DEFERRED — not a current concern).** Menu uses `z-50`, matches existing dialog/combobox/select portals. No observed conflict; leaving as-is per Codex's "leave for Phase 2 unless runtime smoke shows overlap" guidance.

**Nit (1, addressed in-eval):**

- `email.slice(0, 2)` produces odd initials for `@example.com`, `.name@…`, numeric prefixes. Resolution: extracted `deriveInitials()` helper that pulls the first two alphanumerics from the local part, uppercases, falls back to `?` on degenerate input.

### Codex "Not An Issue" (false-positive reduction)

- Server Action call surface: `<form action={signOut}>` inside a client component is the canonical Next 15 pattern. Server Action is RPC'd over the boundary cleanly; no SSR concern.
- Trigger button styling: removing `asChild` and letting `DropdownMenu.Trigger` render its own button preserves the aria-label + Tailwind classes (Radix forwards both); no a11y regression vs the prior pill.
- Server-vs-client boundary: `UserMenu` is `'use client'`, `AppHeader` is server. Only `email: string` crosses — fine.

## Coverage Caveats

- **Open-menu interactive cycle**: the browse tool's MCP a11y tree printer doesn't assign an `@e<n>` ref to the Radix `DropdownMenu.Trigger` button (other buttons on the same page get refs as expected). Likely an MCP quirk for elements with `aria-haspopup="menu"` + `aria-expanded` dynamics. Closed-state verification (trigger visible, email + Sign out absent from the bar) confirmed via screenshot. The actual menu-content rendering, keyboard navigation (Esc, Arrow keys), and Server-Action dispatch are wired per Radix defaults + the canonical `onSelect`+`requestSubmit` pattern; runtime exercise can be revisited when a tooling workaround surfaces (e.g. browse-tool gains keyboard-press support, or a11y-tree printer is updated).
- **Sign-out submission**: post-fix uses the canonical Radix-recommended pattern — `e.preventDefault()` + `requestSubmit()` is well-trodden ground; the race Codex flagged is closed at the React + DOM level. End-to-end sign-out (which redirects to `/login`) wasn't smoke-driven this phase to avoid losing the injected Playwright session mid-test; a manual cycle can confirm if needed before chunk close.
