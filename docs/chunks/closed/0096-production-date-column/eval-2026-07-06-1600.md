## Codex Adversarial Review

**Low**

- Stale `status` query params were preserved and forwarded after the filter was removed. `production-admin.tsx` seeded `URLSearchParams` from the full current URL and only updated `q`/`cancelled`, so `/production?status=past` stayed `status=past` after searching/toggling; `production-page-actions.tsx` forwarded the full query string to `/production/export`, while `export/route.ts` ignores `status`. Old bookmarked URLs / copied export links could still advertise `status=upcoming|past|1m` while table + CSV include all non-cancelled matching rows — contradicting the intent's "no dead time-window params left behind."
  - **Fixed in-cycle at `a4dba6c`:** `pushParams` now `sp.delete('status')` on every navigation, and the export href builder strips `status` too. Functionally the table/CSV were already correct (the route ignores the param); this closes the cosmetic stale-URL gap.

**Not Issues (confirmed by Codex)**

- The re-homed TanStack filter does apply: `columnFilters` targets `id: 'date'`, and the Date column exists with the `filterFn` attached.
- One column carrying `accessorKey: 'startDate'` + `id: 'date'` + sorting + a `filterFn` is valid — sorting uses the accessor value, the filter reads `row.original.status`.
- ISO `YYYY-MM-DD` sorts lexically = chronologically; `campaigns.start_date` is non-null in the schema.
- No remaining live references to the deleted `ProductionRange` / `isProductionRange` / `rangeWindowEndIso` / `campaignTimeStatus` / the removed `status` column id.
