## Codex Adversarial Review

(Verbatim findings; classification + disposition added by the orchestrator.)

### High
None observed.

### Medium

1. **Linked double-submit can still hit a stale `SyncToken` 400** (`dealer-push.ts:83`). Two concurrent pushes of the same *linked* dealer can both read token N; the first rotates QBO to N+1, the second posts N → stale-object 400. Read-before-write narrows but doesn't eliminate the window.
   - **Disposition: PARK (follow-up a).** Inherent QBO optimistic-lock property; only bites under simultaneous pushes of the same dealer (rare for on-demand single-admin use). Fix shape: catch the stale-token error from `updateCustomer`, re-fetch the SyncToken, retry once.

2. **Create-before-guard leaks an orphan QBO Customer when the DB guard loses the race + still reports `created`** (`dealer-push.ts:93`). The QBO Customer is created before the `quickbooks_id IS NULL` guard; if a concurrent push/sync links the dealer in between, the just-created Customer has no local link yet the call reports `created`.
   - **Disposition: PARK (follow-up b — partially by-design).** `intent.md` explicitly accepts the orphan-on-lost-race as a rare duplicate. Codex's added nuance — that the lost race still *reports* `created` — is a minor reporting wrinkle, not a data-corruption bug. Fix shape: inspect `.returning()`; on zero rows, report `skipped`/`already-linked` and optionally deactivate the orphan QBO Customer.

3. **Sparse update cannot clear fields removed in-app** (`dealer-push.ts:49` + `sparse:true`). Omitted fields in a QBO sparse update mean "leave as-is", so clearing a dealer's address/province/email/phone locally won't clear it in QBO.
   - **Disposition: PARK (follow-up c — product decision).** For a v1 "push to populate QBO" flow, not wiping QBO fields from the app is defensible (and arguably safer). Whether the app should clear QBO fields is a product call. Fix shape: on update, send QBO clear semantics for null/empty locals; add clearing tests.

### Low

4. **Missing `SyncToken` falls back to a guessed `'0'`** (`dealer-push.ts:87`, hypothesis). If a Customer read ever omits `SyncToken`, `?? '0'` guesses. QBO always returns it on a read, so hypothetical.
   - **Disposition: PARK (follow-up d).** Fix shape: throw when `current.SyncToken == null` rather than guessing.

5. **Duplicate-name detection only handles string `code: "6240"`** (`client.ts`, hypothesis) + **`safeparse-required` lint** on the Server Action.
   - **Disposition: lint half FIXED IN-CYCLE at `a7d3cc9`** (switched `pushDealerSchema.parse(...)` → `safeParse(Object.fromEntries(formData))` + throw-on-invalid, preserving the gate-matrix allow-with-error semantics). The `String(e.code)` robustness half is PARKED (follow-up e) — QBO returns Fault codes as strings today, and the `/\b6240\b/` text fallback covers JSON-parse failures.

### Explicitly NOT an issue (Codex false-positive reduction, confirmed)
- **Auth gate**: `assertCan('admin:access')` runs before FormData parse, DB load, token refresh, or any QBO write.
- **Gate-matrix harness**: empty-FormData admin path `safeParse`-fails → throws a non-redirect error → harness reads as "allow" (matches the new matrix row, which passed across all 7 roles).
- **Server Action boundary**: mutation is a Server Action, not a route handler (only QBO route handler remains the OAuth callback).
- **`revalidatePath` before `redirect`**; `encodeURIComponent(id)` on the fetch-by-id path; name/address are JSON-serialized for QBO + React-escaped on render (no injection).
