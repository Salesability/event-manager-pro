## Codex Adversarial Review

**Critical/High:** None.

**Medium:**

1. **Duplicate dealer names break keyboard selection identity.** `Command.Item value={o.label}` made cmdk track items by their visible name; two dealerships sharing a display name (e.g. two "Capital Ford" rows) would collide on Arrow+Enter keyboard selection. Mouse selection still uses the closure-captured `o.value`, but keyboard navigation could pick the wrong one. **Status: fixed in-eval.** Switched to `Command.Item value={o.value}` (dealer ID — guaranteed unique) plus `keywords={[o.label]}` so cmdk filters by label substring while preserving identity. Search UX unchanged ("type a dealership name" works); selection identity hardened.

**Low:** None.

**Non-Issues** (Codex-confirmed):
1. Label-based filtering for normal dealer-name search is intentional UX (now via `keywords` after the Medium fix).
2. No stale-closure issue in `onSelect` — each mapped item closes over its current `o.value`.
3. `--radix-popover-trigger-width` is real — installed Popover 1.1.15 sets it from `--radix-popper-anchor-width`.
4. Hidden-input wire format still round-trips: `setDealerLink` → `dealerLinks` state → `<input type="hidden" name="dealerLinks" value="<id>:<role>">` → `parseDealerLinksField` reads repeated entries.
5. Multiple dealer-link rows have independent Popover open-state (each `<Combobox>` instance owns its own `useState(false)`).
6. Radix Select keyboard parity with native `<select>` is covered by the primitive (Space/Enter to open, arrow/home/end to navigate, Space/Enter to select).

Codex notes the verification was source-traced + read-only; browser interaction not run on its end either (sandbox limitation).
