## Codex Adversarial Review

(Verbatim findings; classification + disposition added by the orchestrator.)

### High
None found.

### Medium

1. **Duplicate `create` for two QB customers with the same `lower(name)+lower(address)` and no existing dealer.** Both classify as `create`; both inserts succeed because their QB IDs differ → two duplicate dealer rows. *(`applyDealerSync`, classify computed once from the pre-sync snapshot.)*
   - **Disposition: PARK (by-design per intent).** `intent.md` non-goal: *"if the name+address heuristic doesn't match, we insert; we don't try to be clever about near-duplicates."* Two distinct QB customer records → two dealers is the documented behavior. Very unlikely in practice (QBO IDs are unique; identical name+address dupes are rare — the 0060 import handled its one real case via `DEDUP_NAMES`).

2. **Missing address degrades the match into name-only.** `nameAddrKey` normalizes a null address to `''`, so an address-less existing dealer links to an address-less QB customer of the same name — could attach the wrong QB customer.
   - **Disposition: PARK (inherited from 0060).** The 0060 importer matches on `lower(name)=… AND lower(coalesce(address,''))=…` — identical name-only-when-empty semantics. Not a new regression. Prod dealers seeded by 0060 mostly carry addresses. Hardening (require non-blank address on both sides for `link`) would diverge from 0060's matching and is a follow-up.

3. **Duplicate *existing* dealers with the same name+address → nondeterministic link/collision.** `byNameAddr.set` keeps last-wins; if one same-key dealer is linked to QB `999` and another is unlinked, a customer with a different QB ID may `link` to the unlinked one instead of `skip-collision`.
   - **Disposition: PARK.** Requires a rare DB state (the unique index is on `quickbooks_id`, not name+address; the 0060 import deduped names in prod). The fix (store arrays per key, skip-ambiguous when multiple unlinked match) carries its own product judgment.

4. **Archived dealers participate in matching.** `loadExistingDealers` reads all dealers (no `archivedAt IS NULL` filter); an archived dealer could be linked and shown "already linked" while excluded from normal workflows.
   - **Disposition: PARK (inherited from 0060 + product decision).** The 0060 importer also queries dealers without an `archivedAt` filter. Whether sync should skip / revive / create-new for an archived match is a product decision; sandbox has no archived dealers today.

### Low

5. **`decodeSyncSummary` accepted numeric-prefix junk** — `parseInt('1x',10) → 1`, `parseInt('1e9',10) → 1`, so `?synced=1x.2.3` rendered a bogus summary, contradicting the "rejects malformed params" test.
   - **Disposition: FIXED IN-CYCLE at `f96c140`.** Added an all-digits guard (`/^\d+$/` per segment) before `parseInt`; added `'1x.2.3'` and `'1e9.0.0'` → null test cases. Admin-only, display-only param (no security impact), but the code now matches its test claim.

### Explicitly NOT an issue (Codex false-positive reduction, confirmed)
- The partial-index `onConflictDoNothing({ target, where })` shape is valid for the installed `drizzle-orm@0.45.2` and renders `(quickbooks_id) where quickbooks_id is not null do nothing`.
- Schema + migration partial-index predicates match (`quickbooks_id IS NOT NULL`).
- The `link` UPDATE is correctly guarded (`WHERE id = ? AND quickbooks_id IS NULL` + `RETURNING` to count lost races as skipped) — no clobber of an existing QB id.
- Local `name`/`address` not written on `link`; `province` only set when the snapshot province is null.
- Both entry points admin-gated (`assertCan('admin:access')` on the page and the Server Action).
