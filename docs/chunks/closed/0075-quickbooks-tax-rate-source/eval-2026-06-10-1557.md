## Codex Adversarial Review

### Pass 1 (full chunk diff) — verbatim

**High 1 — Ontario false-match on generic "on".** `codeNamesProvince` matched `\bON\b` case-insensitively, so a code named "GST on sales" / "Tax on purchases" matched Ontario; if the only rate-resolvable match, it linked ON and adopted a wrong rate.

**Medium 1 — broken duplicate codes ignored before ambiguity.** Candidates were name-filtered, then unresolvable rates dropped *before* the ambiguity count, so two "HST ON" codes (one broken) linked the good one instead of flagging ambiguous.

**Medium 2 — empty TaxCode response clears every link.** `applyTaxCodeSync([], rates, tx)` marked all provinces unmatched and cleared every `quickbooks_tax_code_id`; a transient empty read would unmap all provinces and block taxed pushes.

**Low 1 — stale schema comments** in `tax-rates.ts` ("Admin-editable", "matches the province's rate").

**Not issues (false-positive reduction, confirmed correct):** deferred per-province override (by design), kept app-rate fallback for unmanaged provinces, executor never writes NULL into the NOT NULL `rate` column, editor removal mechanically complete (no dangling refs; `queries.ts` correctly retained for quote flows).

### In-cycle fix round 1 — `e98862b`
- **High 1:** matched the abbreviation as a case-sensitive uppercase token (province codes are uppercase; "on" the word is lowercase).
- **Medium 1:** restructured `resolveProvinceLinksByName` to count NAME matches first (>1 → ambiguous) before rate-resolvability.
- **Medium 2:** added a fail-closed guard — throws if `qboCodes` or `qboRates` is empty, before any DB write.
- **Low 1:** updated `tax-rates.ts` schema comments to QB-as-source-of-truth + editor removal.

### Pass 2 (re-review of the fixes) — verbatim summary
- **Medium 1 — FULLY RESOLVED.** Name matches counted before rate resolvability; single-broken-rate path returns `unmatched`. Test-covered.
- **Medium 2 — FULLY RESOLVED.** Correct `|| / === 0` guard, runs before any read/write; legitimate no-province-name clear path still intended.
- **Low 1 — FULLY RESOLVED.**
- **High 1 — PARTIALLY RESOLVED.** The lowercase collision was fixed, but the uppercase word-token still matched an all-caps standalone "ON" mid-name ("GST ON SALES", "ON HOLD") → still mislinked Ontario.
- **New Low (speculative):** the stricter token boundary will miss compact/decorated names ("HSTON", "PEI", "HST ON (13%)") if a company uses those instead of the standard "HST ON" / full name.

### In-cycle fix round 2 — `aecbd9e`
- **High 1 (residual):** anchored the 2-letter code to the **final whitespace-delimited token** of the name. QBO's Canadian convention puts the jurisdiction code last ("HST ON", "GST/PST BC"), so the province form ENDS in the bare code while the preposition form ("GST ON SALES", "ON HOLD") does not. Deterministically verified by unit tests: `'GST ON SALES'`/`'ON HOLD'` → no match; `'HST ON'` → ON. Fails safe — decorated/non-standard names fall through to **unmatched/flagged**, matching the owner's "flag, don't guess" decision.
