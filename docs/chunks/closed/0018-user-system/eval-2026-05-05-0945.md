## Codex Adversarial Review

### High

1. **`promote-admin` can silently split auth-user and contact ownership.**
   - Location: `scripts/promote-admin.ts:72`
   - What: `ensureContact()` returns a contact found by email even if `contacts.user_id` already points at a different auth user. The guarded update at `scripts/promote-admin.ts:74-77` no-ops in that case, but the script still sets `app_metadata.role = 'admin'` on the target auth user and writes/restores `team_member_roles(admin)` for the *other* user's contact.
   - Why it matters: The operator gets success output while the two role sources now describe different people. Future membership reads by `contacts.user_id` will not match the auth user that has the admin gate bit.
   - Suggested fix: Read `contacts.userId` before updating and abort loudly if it is non-null and not the target auth user, before mutating `app_metadata`.

2. **`setUserRoles` can leave `app_metadata.role` inconsistent with `team_member_roles`.**
   - Location: `src/features/auth/actions.ts:213`
   - What: `applyRoleSet()` updates Supabase Auth metadata before the DB transaction that inserts/restores/archives role rows. There is no per-user lock and no compensation if the transaction fails.
   - Why it matters: `app_metadata.role` is the admin gate source while `team_member_roles` is the relationship source. A DB error or two concurrent admins can grant/deny `/admin/users` access independently of the contact role rows.
   - Repro: Admin A submits roles `[coach]`; Admin B submits roles `[admin]` for the same user concurrently. Final auth metadata can be `admin` while the DB transaction from A archives the active admin row, or vice versa.
   - Suggested fix: Do DB sync first, then `app_metadata`. Catch DB sync failures and return controlled `ActionResult`. (Strict per-user serialization via advisory lock is overkill for a 5-person team but worth noting.)

### Medium

1. **`createUser` can leave an auth user behind after role assignment fails.**
   - Location: `src/features/auth/actions.ts:114`
   - What: Auth user is created first, then `applyRoleSet()` runs. If role application fails (e.g., the new user has no contact link yet — currently the common case in Phase 1), the action returns an error but the auth user remains.
   - Suggested fix: Either preflight whether role assignment is possible before creating the auth user, OR (simpler) make the no-contact case a successful "created without roles" state with a clear UI hint that roles will be assignable after Phase 3 contact linkage.

2. **Deactivation archives the contact and breaks active-coach lookup paths for already-assigned campaigns.**
   - Location: `src/features/auth/actions.ts:154`
   - What: `deactivateUser()` archives the linked `contacts` row AND `team_member_roles`. Existing coach lookups require active contact + active coach role (`src/features/schedule/queries.ts:294-313`), so:
     - Sending an assigned-coach confirmation email fails as "No email on file" (`src/features/email/actions.ts:86-88`).
     - Public coach share links 404 because `loadCoaches()` excludes the archived contact.
   - Suggested fix: Re-decide what "deactivate user" means in Phase 1. Two options:
     - **Conservative (recommended):** ban the auth user + archive only `team_member_roles`; leave the `contacts` row active so existing assignments / share links keep working. The contact stops appearing as an *eligible* coach (no active role) but historical references resolve.
     - **Aggressive:** also archive the contact (current behavior). Then add archived-tolerant lookups in `email/actions.ts` and `share/coach/[id]`. More work, more places to remember.

3. **`loadAdminUsers` silently truncates after 1000 auth users.**
   - Location: `src/features/auth/queries.ts:21`
   - What: Hardcoded `perPage: 1000`, single page only.
   - Why it matters: Admins beyond the first page disappear from `/admin/users`.
   - Suggested fix: Page until exhausted, or warn loudly when `data.users.length === 1000`. Defer until needed (current team is <10).

### Low

1. **`promote-admin` has the same 1000-user blind spot** (`scripts/promote-admin.ts:41`). Bootstrap script; same fix as above.

2. **Email/contact matching relies on lowercase convention, not a DB invariant** (`scripts/promote-admin.ts:32`). The `contact_identifiers` schema doesn't normalize `value`, so a pre-existing uppercase email row wouldn't match. Suggested fix: lowercase at write boundaries and/or add a normalized-email DB invariant. Hypothesis Codex flagged but didn't verify: Supabase Auth normalizes email; this repo's contact identifier table does not.

### False positives confirmed (Codex actively ruled these out)

- Page + 3 Server Actions are gated at entry (`requireAdmin()` is the first call in each).
- Admin gate trusts `app_metadata` (server-controlled), not `user_metadata` (user-editable).
- No service-role import leaks into client bundles — `users-admin.tsx` only imports the type `AdminUserRow` and the `'use server'` actions; `'server-only'` is on `admin.ts` and `queries.ts`.
- The `team_member_roles_contact_id_role_unique` index is a *full* unique index, so multiple archived rows for the same `(contact, role)` cannot exist — the restore-or-insert logic is correct.
- Historical campaign display in `loadCampaigns()` is unaffected by archive — coach names still render via the unfiltered left-join.
- `parseRolesField` correctly rejects forged `roles=staff` / `roles=viewer` form fields.

### Test coverage gaps Codex flagged

- `setUserRoles`, `deactivateUser`, `loadAdminUsers` have no direct tests; the action test mock is shallow (no real DB behavior). Highest-risk paths (restore/archive/insert ordering, metadata-DB drift, deactivation side effects) are uncovered.
- Query tests don't cover archived-contact rendering in `/admin/users`.
- Service-role boundary is not build-verified — it's enforced by `'server-only'` at runtime/build, but the tests mock `@/lib/supabase/admin` so a regression where `queries.ts` value-imports leak into a Client Component wouldn't fail unit tests (would fail `next build`).
