## Codex Adversarial Review

### Critical

- **Contact-only users can reach the staff app after callback** (`src/lib/supabase/middleware.ts:49`)
  - `/auth/callback` correctly sends users with no `team_member_roles` to `/auth/auth-error`, but the actual app routes only require a valid Supabase session. Middleware allows any authenticated non-`/admin` path, and `src/app/(app)/layout.tsx:8` only checks `getUser()`.
  - A dealer-contact-only or unprovisioned auth user gets a valid session during callback, lands on the error page, and can manually request `/calendar`, `/production`, or `/lists` to render staff data.
  - **Repro:** sign in as a contact with no active `team_member_roles`; callback hits the error branch at `src/app/auth/callback/route.ts:55`; then GET `/calendar` — middleware skips the unauthenticated redirect because `user` exists, layout passes, calendar loads.
  - **Fix sketch:** add a staff-app gate outside callback — either inside `(app)/layout.tsx` (check membership and redirect non-staff to `/auth/auth-error`) or as a middleware-side check that any `(app)/*` path requires either `app_metadata.role === 'admin'` or at least one active `team_member_roles` row (the latter needs a DB hit per request — keep the page-level check unless that becomes a bottleneck).

  **Eval note:** the plan's "v1 wired roles" decision says *"the default for any signed-in non-admin (no `team_member_roles` row) IS the staff experience"*, which conflicts with Phase 5's three-branch routing intent. This finding is the missed durability of that decision — Phase 5 routes the *first* request after callback but doesn't gate subsequent direct URL access. Worth resolving before the portal opens (it's load-bearing then).

### High

- **Contact linkage updates can overwrite a concurrent link** (`src/features/auth/actions.ts:243`)
  - `linkUserToContact` pre-checks that `contacts.user_id` is null, then later updates by `id` only. The `createUser` picked-contact path has the same pattern (`actions.ts:126` pre-check → `actions.ts:167` update).
  - **Repro:** two admins both read `contacts.id=42, user_id=NULL`; B runs `linkUserToContact(userB)` → `user_id=userB`; A then reaches the unconditional `UPDATE` and overwrites `user_id=userA`.
  - **Fix sketch:** make the update conditional — `WHERE id = ? AND user_id IS NULL AND archived_at IS NULL RETURNING id`, treat zero-rows-returned as a conflict. Or wrap the read+write in an explicit transaction with `SELECT ... FOR UPDATE`.

### Medium

- **`createUser` can report failure after creating a durable auth user** (`src/features/auth/actions.ts:140`)
  - Auth user is created before contact linkage and role sync. Several later branches return errors without compensating: auto-link conflict at `:160`, contact-create failure at `:185`, role-sync failure at `:196`.
  - **Effect:** admin sees "user creation failed", but `auth.users` already has the row (possibly auto-linked by trigger). Retry yields a duplicate-email error.
  - **Fix sketch:** on post-create failure either delete the auth user (`auth.admin.deleteUser`) for a clean retry, or surface explicit partial-success ("auth user created but contact link failed — link manually") and refresh the list.

### Low

- **Tests miss the new linkage and predicate-sensitive failures** (`src/features/auth/actions.test.ts:38`)
  - The DB mock ignores `WHERE` predicates and shifts queued arrays positionally, so tests can pass even if production drops `archived_at`/`user_id` guards. Callback tests have the same shape.
  - The `createUser` suite stops at unsupported-role rejection; no happy-path coverage for `contactId` pick or firstName/lastName create-new-contact.
  - **Fix sketch:** in a follow-up, add tests asserting conditional update predicates (RETURNING zero rows on conflict), trigger-conflict behavior, and the createUser linkage paths end-to-end. For callback tests, capture the `contactId` and `archived_at` predicate the dealer-contact lookup ran with.

### Not an issue (false-positive reduction)

- **`/auth/callback`'s `next` parameter is not the bypass.** The route checks membership before redirecting to `next`; only `roles.length > 0` reaches `new URL(next, url)`. The Critical finding above is about *post-callback* direct access, not `next=/calendar` during callback.
- **`applyRoleSet` DB-first, then `app_metadata` is by design.** If the auth update fails, admin gate stays *closed* — stale gate cache, not privilege escalation. The plan's Decisions section explicitly accepts hybrid drift, and the action returns "Roles saved, but the auth gate did not update".
- **No new raw SQL interpolation.** All new TypeScript paths use Drizzle helpers (`eq`, `and`, `inArray`, `isNull`). The hand-written migration is the only new SQL.

## Cross-Reference

- **Codex Critical (post-callback access) ↔ smoke gap.** The web-test couldn't fully verify the callback flow as a contact-only user because no such fixture exists in dev. The Critical finding identifies the issue conceptually; a destructive-fixture smoke would need to provision a fake contact-only auth user, sign in as them, then attempt direct staff-route access. Out of scope for `/eval` — file as a follow-up smoke.
- **Codex High (TOCTOU on link updates) ↔ Codex Low (test mock predicate-blindness).** Same root cause: the predicate is enforced in production code only on the read, not on the write, AND the test mock can't see it either. Tightening the production predicate (conditional UPDATE with RETURNING) and updating the test mock to capture predicates would close both.
