## Codex Adversarial Review

Codex CLI job `task-mrdo2k36-qychhm`, verbatim:

> **High**
> None found.
>
> **Medium**
> - `src/app/(app)/dealerships/[id]/page.tsx:73` — `returnEvent` is not dealer-scoped before it drives the MSA return flow.
>   Repro: visit `/dealerships/2?returnEvent=<campaignId-for-dealer-1>` as an admin. The page loads that campaign by id only, shows a Back-to-event link for it, passes the id into the MSA dialog, and after sending an MSA for dealer 2, `msa-create-dialog.tsx:77` redirects to dealer 1's event.
>   Why it matters: a tampered or stale URL can make the workflow hub return to the wrong event after a real MSA send. The fix should validate `returnEventCampaign?.dealerId === dealer.id` before showing the link or passing the id to the dialog.
>
> - `src/features/schedule/next-step.ts:33` — next-step emphasis can disagree with the real accept gate for active MSAs with `expiresAt = null`.
>   Repro: an `active` MSA row whose `expires_at` is null. `effectiveMsaStatus` treats that as active (`commercial-status.ts:72`), so a sent quote gets `accept-quote` (`next-step.ts:44`). But `acceptQuote` requires `expires_at >= now()` (`accept-gate.ts:58`), where null fails.
>   Why it matters: the event dialog can emphasize "View Quote" as the accepting path even though the server will reject acceptance. Current signing lifecycle appears to stamp `expiresAt`, so this is mainly legacy/manual/backfill data, but the schema allows it.
>
> **Low**
> - `src/app/(app)/calendar/calendar-view.tsx:135` — unresolved or cleared `?event=` does not close an already-open dialog.
>   Repro: client-side navigate from `/calendar?event=42` (open) to `/calendar?event=999999` or `/calendar`. The reconciliation only calls `setDialog` when a campaign is found; for null/unresolvable targets it just latches or does nothing, leaving the previous event dialog visible.
>   Why it matters: the URL can claim one event, or no event, while the UI still shows the previous event. I do not see an infinite-render risk in the guard.
>
> **False Positives / Not Issues**
> - Calendar `?event=` parsing rejects empty, negative, non-integer, `NaN`, and huge values via `Number.isInteger(n) && n > 0`. Array params use the first value consistently.
> - The render-time `setState` pattern in `CalendarView` is guarded by `reflectedEventId !== initialEventId`; for resolvable ids it settles after one reconciliation render.
> - The added `Quote.campaignId` projection is wired through the `Quote` type, projection, row type, and mapper consistently.
