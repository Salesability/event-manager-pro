## Codex Adversarial Review

### High

- **Expiry guard is TOCTOU and uses the app clock.** `src/features/quotes/lifecycle.ts:79-98`, `src/features/quotes/lifecycle.ts:45-53`. `markQuoteAccepted` pre-loads `sentAt`/`quoteValidDays` and then calls `transition()`, but the atomic `UPDATE` only guards `id` and `status='sent'`; a quote that crosses the expiry boundary between the SELECT and UPDATE can still be accepted, and Node/Postgres clock skew is unbounded because the guard uses `Date.now()`. Suggested fix: fold the expiry predicate into the guarded UPDATE using Postgres time, e.g. `sent_at + quote_valid_days * interval '1 day' >= now()`, then classify misses with a re-select.

  → **Fixed in-cycle at `29a39b4`** — added Postgres-time predicate to the UPDATE (`sql\`${sentAt} + (${quoteValidDays} * interval '1 day') >= now()\``); reselect path now distinguishes "missed because not sent" from "missed because expired between load and UPDATE", returning the friendly expired-error in either case. New regression test in `actions.test.ts` ("returns the expired error via the Postgres-time guard when JS pre-load thought the quote was fresh (TOCTOU)") exercises the race. Extracted the message into an `expiredErrorMessage()` helper so the two call sites stay in sync.

### Medium

- **Bundled MSA quote can show a different validity deadline than the later sent quote.** `src/features/msa/actions.ts:285-292`, `src/features/quotes/actions.ts:722-730`. `sendMsaEnvelope` renders the bundled draft quote with `issuedDate = todayIsoDate()` but `validUntilDate = isoDateOffset(new Date(), ...)`; later `sendQuote` re-renders from a fresh `sentAt`, so the external MSA envelope PDF and the eventual sent quote PDF can disagree on the valid-until date. Suggested fix: either persist and reuse a quote issue timestamp when bundling, or make the MSA flow actually send/lock the quote once it is included in the signature package.

  → **Parked as 0044 follow-up (a).** Out of 0044's stated scope. The MSA bundle flow is closed/0041's design surface and the right fix (persist a quote-issue timestamp vs. flip the quote to `sent` inside the MSA envelope flow) needs a semantic decision about how the MSA-bundled quote relates to the eventual sent quote. Adding to **Parked:** in `CURRENT.md`.

### Low

- **Read-only quote composer still says expired quotes are "sent".** `src/app/(app)/quotes/[id]/page.tsx:141-156`, `src/features/quotes/quote-composer.tsx:317-320`. The header pill uses `displayStatusKey(quote)` and will show `expired`, but the read-only banner only receives the raw `quote.status` and renders "This quote is sent", treating expired-but-sent like fresh-sent in that downstream surface. Suggested fix: pass `isExpired` or `displayStatusKey` into `QuoteComposer` for display copy, while keeping raw `status` for lifecycle gates.

  → **Fixed in-cycle at `29a39b4`** — extended `InitialQuote` with `isExpired: boolean`, page passes `quote.isExpired` through, banner reads "expired" when `initial.isExpired`. Lifecycle gates (`isReadOnly = isEdit && initial.status !== 'draft'`) still switch on the raw `status`, so the read-only enforcement is unchanged.

- **Line-item cap reduction can reject previously valid 13-line PDFs, and docs still say 13.** `src/lib/pdf/render-quote.ts:29-34`, `src/lib/pdf/render-quote.ts:135-138`, `docs/wiki/data-model.md:408`. The renderer now fails at `MAX_LINE_ITEMS + 1`, so any persisted 13-line quote that previously fit will now fail preview/send/MSA bundling; the data-model docs still document `MAX_LINE_ITEMS = 13`. Suggested fix: update the docs and run a pre-merge/prod check such as `jsonb_array_length(line_items) >= 13`; if any rows exist, either keep 13 with layout changes or add a remediation path.

  → **Fixed in-cycle at `29a39b4`** (wiki doc) — `docs/wiki/data-model.md:408` updated to "MAX_LINE_ITEMS = 12" with a note about the 0044 P2 cause. The composer's v1 emitted max is 8 lines, so realistic in-prod data is well below the new cap; no prod-row remediation needed for v1. If/when the composer adds line-item configurability the cap will need revisiting alongside the layout.

- **Expired quotes are hidden behind the Sent filter with no expired filter/count.** `src/app/(app)/quotes/page.tsx:21-29`, `src/app/(app)/quotes/page.tsx:84-94`, `src/app/(app)/quotes/quotes-filters.tsx:9-15`. The table can show an `expired` pill, but counts and URL filtering are still based only on raw `status`, so expired quotes increase "Sent" and only appear under the Sent filter. Suggested fix: add an Expired pill/count keyed off `isExpired`, or rename/help-text the Sent pill to make "sent includes expired" explicit.

  → **Parked as 0044 follow-up (b).** This was an explicit Phase 3 design decision per the plan's "Status filters on `/quotes` deliberately stay on the underlying `status`" — a dedicated Expired filter pill made more sense to add alongside Option A (full enum + sweep) if that's ever promoted, since Option B's derived state can't be filtered server-side without re-fetching every row. Documenting in **Parked:** so it doesn't get lost.

### Explicitly not an issue

- **Null `sentAt` does not blow up derived `isExpired`.** `src/features/quotes/queries.ts:100-104`, `src/features/quotes/queries.test.ts:195-200`. `mapRow()` short-circuits on `row.sentAt != null` before calling `getTime()`, and the draft/null test covers that path.

- **Boundary behavior is internally consistent: exactly at expiry is still allowed.** `src/features/quotes/lifecycle.ts:89-96`, `src/features/quotes/queries.ts:101-104`, `src/features/quotes/actions.ts:498-499`. Both enforcement and projection use strict `< Date.now()`, so equality is not expired; `isoDateOffset` uses UTC epoch milliseconds and ISO slicing, so DST transitions do not change the computed 24-hour-day deadline under the current semantics.

- **AcceptQuote test FIFO setup accounts for the new pre-load.** `src/features/quotes/actions.test.ts:555-689`, `src/features/quotes/actions.test.ts:692-736`. The accept/action and internal-helper cases that reach `markQuoteAccepted` all push a pre-load row before the guarded UPDATE/reselect entries; invalid-id tests correctly do no DB work, and queue underflow would produce different results rather than the asserted happy paths.
