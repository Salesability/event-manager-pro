## Codex Adversarial Review

### Medium (fixed in-cycle at `1c6b28c`)

- **`divide-stone-100` survived the migration** in 5 callsites: `src/components/ui/data-table.tsx:189`, `src/features/quotes/quote-composer.tsx:642`, `src/features/schedule/availability-admin.tsx:97`, `src/features/schedule/lookup-admin.tsx:146`, `src/features/services/services-admin.tsx:84`. Audit + codemod covered `border-stone-*` but not `divide-stone-*`. **Fix:** Extended `codemod.mjs` with `divide-stone-{100,200} → divide-border`, `divide-stone-300 → divide-input`. Ran across the 5 files. Zero residual `divide-stone-*` after fix.

### Low (fixed in-cycle at `1c6b28c`)

- **Stale token-language comment** at `src/features/people/people-admin.tsx:116-123` — comment still said the role checkbox "flip[s] the navy-on-white look" though the actual checked classes are `data-[state=checked]:border-primary data-[state=checked]:bg-primary`. **Fix:** Refreshed comment to "flip to primary-fill on `data-state="checked"`".

### Ruled out (with evidence — preserved verbatim from Codex)

- **Legacy raw CSS vars in live code** → no confirmed live `var(--color-stone-*)`, `var(--color-navy*)`, `var(--color-cream)`, or `var(--font-display)` users found; `globals.css` now exposes semantic aliases plus kept palette/status tokens only. The remaining `--color-accent` is the semantic Tailwind alias, not the removed legacy `--color-accent` brand layer.
- **`Button` prop pass-through / submit semantics** → the shadcn `Button` spreads remaining props onto the Base UI primitive (`src/components/ui/button.tsx:43-53`); quote composer action buttons explicitly keep `type="button"`, the people form submit button explicitly keeps `type="submit"`.
- **`buttonVariants` export/import shape** → `buttonVariants` is exported with `Button` (`src/components/ui/button.tsx:58`) and imported from the same module in `people-admin`.
- **`DialogClose` className wiring** → local `DialogClose` forwards props directly to the Base UI close primitive (`src/components/ui/dialog.tsx:22-23`), so `className={buttonVariants(...)}` is not clobbered by an internal wrapper.
- **Filter chip vs. Badge scope** → `pillClass` is attached to actual `<button>` controls with `aria-pressed` and filter toggling handlers (`src/features/people/people-admin.tsx:143-147, :244-270`), so keeping custom interactive filter-chip chrome instead of a status `Badge` is defensible.
- **Public login/share branding** → both public surfaces still carry brand signals through the real logo and primary header/action color after `bg-cream` moved to `bg-background`.
- **Print stylesheet** → the print block uses literal `#fff`/`#000` and generic element rules, with no legacy custom-property references in the print path.
