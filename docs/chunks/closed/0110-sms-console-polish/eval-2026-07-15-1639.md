## Codex Adversarial Review

**High**

None found.

**Medium**

**Synchronous classifier can consume the webhook response budget**
`src/app/api/twilio/webhook/route.ts:183`, `src/lib/ai/classify-sms-thread.ts:45`

After `captureInboundMessage` commits the inbound, the route still awaits `classifyThreadFromInbound` before returning `200`. The Anthropic client timeout is 8s, but that is only the model-call budget; the same request already spent time reading/verifying the body, resolving/creating the thread, inserting the message, selecting the transcript, and later updating the thread. A slow Supabase connection plus an Anthropic timeout can push the Twilio request toward or past Twilio's 15s window. The `try/catch` prevents a thrown classifier error from changing the response status, but it does not prevent late acknowledgement or route-worker tie-up.

Fix shape: return the Twilio `200` immediately after capture and move classification to a background job / queue / platform `waitUntil` equivalent, or race classification against a much smaller remaining webhook deadline and abandon it without delaying the response.

**Low**

**Turn-state can become stale if outbound reply persistence is interrupted between message insert and thread timestamp update**
`src/lib/sms/conversations.ts:253`, `src/lib/sms/conversations.ts:292`, `src/features/sms/conversations/queries.ts:54`

`sendThreadReply` inserts the outbound `sms_thread_messages` row, calls Twilio, and only then updates `sms_threads.last_message_at`. The new `awaitingReply` derivation trusts `last_inbound_at >= last_message_at`. If the process is killed, the request times out, or the final thread update fails after the outbound row is inserted, the conversation can visibly contain a staff reply while `last_message_at` remains equal to the prior inbound timestamp, so the UI continues showing "awaiting your reply."

Fix shape: bump `last_message_at` in the same DB unit as the outbound message insert before the external Twilio call, or derive turn-state from the latest message row instead of the denormalized thread timestamps.

**STOP can leave stale "hot/warm/cold" labels visible on an opted-out thread**
`src/app/api/twilio/webhook/route.ts:171`, `src/features/sms/conversations/inbox-thread-list.tsx:53`, `src/features/sms/conversations/conversations-panel.tsx:169`

The classifier only runs in the non-STOP branch. A thread previously classified as `hot` can later receive `STOP`; `captureInboundStop` appends the STOP and bumps `last_inbound_at`, but the old `prospect_temperature` remains and both inbox and conversation rows render the sentiment/temperature before the opt-out badge. The resulting row can show "hot prospect" and "opted out" together even though the latest customer message is a hard stop.

Fix shape: on STOP, clear the display labels or stamp a deterministic cold/negative classification, or suppress sentiment/temperature UI for opted-out threads.
