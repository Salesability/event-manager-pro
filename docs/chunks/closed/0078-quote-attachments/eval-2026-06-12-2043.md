## Codex Adversarial Review

7 findings. Severity verbatim, with disposition.

### Fixed in-cycle (`59c01a0`)
- **Medium — Terminal quotes can still have attachments removed.** `removeQuoteAttachment` had no accepted/declined guard (asymmetric with `uploadQuoteAttachment`). **Fixed:** added the same terminal-status guard + a unit test.
- **Low — Display filename not hardened for the email context.** `cleanDisplayFilename` stripped path separators but not control/format chars (CR/LF, bidi). **Fixed:** `cleanDisplayFilename` now strips `\p{Cc}\p{Cf}` before the name reaches the email layer.

### Parked as 0078 follow-ups
- **(a) High — in-flight upload omitted from the sent email but persisted.** A slow upload racing a direct `sendQuote` call: the send snapshots attachments before the upload's insert, emails PDF-only, then the row lands on the now-`sent` quote. **Mitigated:** the UI disables Send while `attachmentPending`, and a re-send includes the persisted file (self-healing). Reachable only by bypassing the dialog (direct action call) — low-probability for a single-admin tool.
- **(b) Medium — attachment changes not covered by the send optimistic lock.** Removing an attachment between `sendQuote`'s snapshot and the guarded UPDATE can email a doc the user "removed before send." Same UI gate; narrow race.
- **(c) Medium — parallel uploads can bypass the 20 MB total cap.** Read-then-write cap check without a quote lock; concurrent direct uploads each see `existingTotal=0`. The send-time guard still fails closed (DB/storage grow but nothing over-size goes out). UI uploads sequentially.
- **(d) Medium — server MIME validation trusts caller `File.type` only** (no magic-byte/extension check). The file is emailed (not executed/served) and the uploader is an authenticated trusted coach — defensible v1 limitation.
- **(e) Low — GCS object orphaned if the DB insert throws after `putObject`.** Retention is keep-forever anyway, so an orphan is harmless storage; `recordAudit` is best-effort (never throws).

### Explicitly NOT issues (Codex-confirmed)
- GCS key traversal — `sanitizeAttachmentFilename` takes the basename + collapses to `[A-Za-z0-9._-]`; key stays under `quotes/{id}/attachments/`.
- Same-name collision — `randomUUID()` prefix.
- Remove IDOR — delete predicate is `id AND quoteId`.
- Raw-vs-base64 cap — 20 MB raw leaves headroom under Resend's ~40 MB after ~33% inflation.
- Missing GCS object at send — `getObject` failure returns **before** the status transition (fails closed).
