## Codex Adversarial Review

**High** — none.

**Medium**
1. `schedule/queries.ts` — overdue uses server-UTC "today", not viewer-local, contradicting
   decision.md D3's "agrees with the queue" claim. Concrete: at 21:30 America/Montreal the UTC
   date is the next day, so a dealer due today is counted overdue by the dashboard but not the
   queue. **→ FIXED in-cycle** (`6abfac7`): reconciled decision.md D3 + the loader comment to
   describe the server-UTC behavior + its late-evening boundary caveat; the queue stays the
   authoritative per-viewer view (a management overview only differs by ≤1 day after the UTC
   rollover, outside business hours).
2. `pipeline-dashboard.tsx` — the "Stale" card drilled through to `?idle=1`, but the queue's
   `idle` filter is `isIdle(nextAction)` (no *commitment*), a different set than stale
   (`isStale(lastContactedAt)` — no *touch*). Clicking "Stale" would exclude genuinely-stale
   dealers that have a next-action set. **→ FIXED in-cycle** (`4ae9f23`): the card header now
   links to the unfiltered prospect queue (`?status=prospect`); the per-row links still
   deep-link the exact stale dealer. decision.md drill-through table updated.

**Low**
1. `pipeline-dashboard.tsx` — blocker copy said "21+ / 14+ days" but the predicates are strictly
   `>` (exactly 21/14 days is not flagged). **→ FIXED in-cycle** (`4ae9f23`): copy → "Over 21
   days" / "Over 14 days".
2. `schedule/queries.ts` — the 30-day `dealer_activities` window (`gte(occurred_at, since)`)
   has no leading-`occurred_at` index (the only index is `(created_by_id, occurred_at)`), so the
   rolling read can degrade to a scan as activity volume grows. **→ PARKED as 0088-a** (needs a
   migration; `dealer_activities` is tiny today — 0087 just shipped — so no v1 urgency).
