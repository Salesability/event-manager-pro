## Codex Adversarial Review

Read-only review against `/tmp/eval-0045-diff.patch` (3916 lines, HEAD~16..HEAD).

### High

_None._

### Medium

1. **`createPerson` / `updatePerson` — appAccess + email='' on already-linked user.** `src/features/people/actions.ts:560`. The guard `if (appAccess && !email && current.userId == null)` only fires when the user has no auth-side link yet (off→on path). An admin submitting `appAccess=1` with `email=''` against an already-linked person bypasses the check; the transaction then archives the primary email identifier via `swapPrimaryIdentifier(..., '')` while the auth-side sign-in email is preserved, leaving the contact email-less while still able to log in. **Pre-existing bug, not a 0045 regression** — `updatePerson` had this shape before the chunk. Parked under 0045 follow-ups.

2. **`dealerFormSchema` status — empty string rejected.** `src/features/dealers/dealer-schema.ts:31`. After Phase 2 dropped the original `z.preprocess(blankToUndefined, ...)` (lost during TS type-inference friction with `zodResolver`), a programmatic FormData with `status=''` on the update path would fail `safeParse` instead of preserving the existing DB status as the action's comment promised. **In-chunk regression — FIXED in-cycle at `7c535b4`** by adding a `normalizeDealerWire` wire-format pre-pass that deletes `status=''` before `safeParse`. Regression test landed at `dealers/actions.test.ts:208`. The form path is unaffected (the form's `valuesToFormData` always emits a definite `status` value).

3. **ESLint rule policy-strength gap.** `eslint-plugins/safeparse-required.mjs:65`. The rule accepts any reachable `safeParse` call regardless of (a) whether the argument is `Object.fromEntries(formData)`, (b) whether it runs before mutation, or (c) whether the call sits in a dead/unrelated branch. An adversarial action could insert/update from raw `formData`, then call `someSchema.safeParse({})` later and pass lint. **Policy-strength gap, not a regression.** Parked under 0045 follow-ups — a future hardening pass could inspect the call argument shape + enforce "before mutation" ordering. The current shape catches the common drift (no schema usage at all), which is what the 0045 doctrine targets.

### Low

_None._

### Ruled out (false-positive reduction)

Codex explicitly ruled out each of these focus-area concerns:

- Repeated `roles` / `dealerLinks` collapse via `Object.fromEntries` — N/A; `personFormSchema` excludes them and `people/actions.ts` keeps `formData.getAll(...)` for those keys.
- `acquiredVia` preserve-vs-clear semantics — preserved; `updateDealer` uses `formData.has('acquiredVia')` to discriminate absent vs `''`.
- Quote `validation: skip` opt-outs (`setQuoteTax`, `setQuoteDealer`, `acceptQuote`, `declineQuote`) — confirmed no non-trivial unvalidated fields; `parseTax` / `parseId` cover the IDs.
- `parseCampaignInput` error ordering — schema field-order makes missing `startDate` surface first, matching the legacy single-string error tests.
- Cross-field availability errors — single-string `{ error }` flow preserved for the existing UI toast.
- Service-item money normalization — `toServiceItemFields` calls `normalizeMoney` in every code path (insert, un-archive UPDATE, edit UPDATE); `MONEY_RE` correctly rejects `2.675`.
- `toLegacyResult` `fieldErrors` extension — only attached to error results, so `'ok' in result` branching is unaffected.
- Imported-wrapper lint bypass beyond `parseCampaignInput` — `eslint.config.mjs` only configures that one wrapper name; same-file detection doesn't trust arbitrary imported names.
