## Codex Adversarial Review

**Critical:** none.
**High:** none.

**Medium**
- `src/features/schedule/queries.test.ts` is too predicate-blind. The mock accepts almost any `select().from().join().groupBy().orderBy()` shape and only verifies the final mapper. A query could drop `COALESCE`, use the wrong join, omit `groupBy`, group by the wrong column, or change `count(${campaigns.id})::int`, and these tests would still pass. This also does not satisfy the plan's "fixture corpus" intent. Use a real test DB fixture or, at minimum, assert the generated Drizzle SQL/query shape.

**Low**
- `::int` casts are correct for normal app-sized data, but they make overflow a hard SQL error if `count()` or `sum(integer)` exceeds signed int range. Postgres `sum(integer)` returns `bigint`; `coalesce(..., 0)::int` forces it back to int4. Probably fine, but if imported legacy data can accumulate huge `qty_records` totals, prefer `::bigint` plus explicit JS conversion/formatting.
- `loadCampaignsByCoach()` returns `groupKey: null` for "Unassigned". Phase 2 table/filter/export code should handle nullable row IDs deliberately, especially if keys are used as React row ids, CSV stable identifiers, or faceted filter values.

**Explicitly Not An Issue**
- No SQL injection: `sql<number>` templates interpolate Drizzle column objects + static SQL only.
- `COALESCE(SUM(nullable_int), 0)` semantics correct.
- Postgres `::int` cast is real (not just a TS hint) ‚Äî wire type honored.
- Coach join is 1:1 (`campaigns.coachId ‚Üí contacts.id`) ‚Äî no row duplication risk.
- Month bucketing is timezone-safe (`date` column + `to_char` is TZ-independent; noon UTC fixes JS month-shift).

Auth note: these loaders have no internal gate; Phase 2 must keep `/reports` and `/reports/export` staff-gated. Not a Phase 1 bug.
