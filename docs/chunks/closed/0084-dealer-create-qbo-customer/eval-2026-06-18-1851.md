## Codex Adversarial Review

**High:** None.

**Medium — Cleared app contact fields are omitted from sparse QBO updates, leaving stale QuickBooks data** (`src/lib/quickbooks/dealer-push.ts:51`, `src/lib/quickbooks/client.ts:360`).
`mapDealerToCustomer` only sets `GivenName`/`FamilyName`/`PrimaryEmailAddr`/`PrimaryPhone` when the value is truthy (`if (dealer.contactFirstName) …`). `updateDealer` can *clear* email/phone in the app (the form submits empty strings; `swapPrimaryIdentifier` archives the identifier when `!newValue`), so on reload `loadDealer` returns `null` and the mapper omits the field. Because the QBO update is `sparse: true`, an omitted field is *preserved*, not blanked — so clearing a contact field (or removing the contact link → null name) in the app does NOT clear it in QBO. Same shape for address/province. This is a partial gap in app-as-source-of-truth: the common *change-to-a-new-value* case propagates correctly, but *clear-to-empty* does not.
**Classification:** NOT an in-chunk regression — the omit-on-empty mapper shape predates 0084 (0070); 0084 only newly *auto-invokes* the push, surfacing it on edits, and newly extends it to `GivenName`/`FamilyName`. The proper fix requires deciding the QBO sparse-*clear* shape (how to blank `PrimaryEmailAddr`/`GivenName` in a sparse update — a real QBO-API contract question that can regress if done blind). **Parked as 0084-a.**

**Low — Best-effort QBO push blocks the Server Action response on QBO latency** (`src/features/schedule/actions.ts` helper awaited at the create/activate/edit call sites).
The DB write commits before the push and errors are swallowed (rollback behavior correct), but the action still awaits token refresh + QBO fetch/create/update + backfill before `revalidatePath` + `{ ok: true }`, so a slow QuickBooks makes "Save dealer" appear slow.
**Classification:** matches the established **0077 calendar best-effort pattern** (also awaits `reconcileCampaignCalendar` inline) — a deliberate, consistent trade-off, not a defect. A durable outbox / bounded timeout is the upgrade path. **Parked as 0084-b.**

**Explicitly NOT issues (confirmed by Codex):** active-only create holds; D2 (active OR linked) holds; D1 (6240 → swallow → stays unlinked, no bare-name auto-link) holds; create-branch backfill stays guarded (`WHERE quickbooks_id IS NULL`); update branch reads a fresh `SyncToken`; reload-after-update filters `archivedAt IS NULL` so a concurrent archive isn't pushed; no new exported Server Action (helper is private, runs under the existing `dealer:create`/`dealer:edit` gates).
