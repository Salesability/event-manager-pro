## Coverage Caveats

- **Browser smoke deferred to Phase 4.** Phase 2 is gate-call swaps; the user-visible surface is unchanged. The plan schedules the smoke checks under Phase 4 after the availability action migration lands.
- **Layout swap deferred (refines plan body).** The plan's Phase 2 body says "swap `requireStaffAccess()` for `assertCan('app:access')` in `(app)/layout.tsx`." Phase 2 instead **refactored `requireStaffAccess` to delegate its predicate to `can(profile, 'app:access')`** but kept the layout calling `requireStaffAccess`. Reason: `assertCan` redirects to `/` on deny, but `/` is inside `(app)/` and would loop on a dealer-only contact typing `/calendar` directly. `requireStaffAccess` retains the friendly auth-error redirects (Portal-not-yet-available / Account-not-provisioned). The capability is canonical (the same `can()` decision); only the redirect-target wrapper stays. This refines plan Open Question #1's working assumption and is the same shape — keep specialty redirect helpers as thin wrappers — applied to `requireStaffAccess` rather than just `require-role.ts`.
- **`requireRole` not yet retired.** Phase 3 still references it from `src/features/schedule/actions.ts` (3 sites — `roleListClient(['admin','coach'])` migration to `capabilityClient('availability:edit')`); Phase 4 cleans up wiki references and the `require-role.ts` decision (delete vs keep — Open Question #1).
