## Codex Adversarial Review

**High**

**Medium**

**Low**

[src/components/app/confirm-dialog.tsx:66](../../../src/components/app/confirm-dialog.tsx): `useConfirm()` has no cleanup path for an outstanding promise when the component unmounts. The promise resolver is stored only in React state (line 61) and is only called from the dialog `onResolve` closure (line 76). Concrete trigger path: open any migrated confirm, then navigate away with browser/app navigation before choosing Cancel/Confirm. In `PersonForm`, that can leave the `useActionState` async action (people-admin.tsx:391) awaiting forever after it calls `await confirm(...)` (line 401). No normal in-dialog click/Escape path causes this; this is specifically an unmount/navigation edge.

[src/components/app/confirm-dialog.tsx:66](../../../src/components/app/confirm-dialog.tsx): the hook is explicitly single-slot, and a second `confirm()` call before the first resolves overwrites the first resolver without resolving it (last request wins, line 68; only the current `request.resolve` is called, line 77). Concrete trigger path: a re-entrant same-hook call, such as automated/keyboard activation firing two actions in one event turn on shared-hook surfaces like `SmsPanel` (onImport/onLaunch) or `EventDetail` (onCancel/onEmailClient/onEmailCoach share the hook). Not verified as a normal human double-click path — React discrete updates should commit the modal before the next click reaches the underlying control; treat as a defensive robustness finding, not a proven everyday UI break.

**Not an Issue**

- people-admin.tsx:391 — the confirm inside `useActionState` does render while the action is pending; the confirm's own buttons are independent `type="button"` controls, so they stay clickable while the submit button is disabled.
- people-admin.tsx:174 / dealers-admin.tsx:259 — the memoized column closures introduce no confirmed stale-data bug (row object passed at click time from `row.original`; captured `confirm`/`startTransition`/router/setters are stable).
- alert.tsx:31 — the `z-50` Alert change does not regress other Alert usages (`Alert` is only consumed by the new confirm component) and matches Catalyst Dialog's z-40 backdrop / z-50 panel.
- confirm-dialog.tsx:36 — Escape/backdrop cancel parity intact (Catalyst Alert passes `onClose` through to Headless Dialog; the confirm maps close → `onResolve(false)`); buttons are `type="button"` so the People-form confirm does not submit the underlying form.
- All 10 migrated gates still abort on cancel: each rewritten handler checks `if (!(await confirm(...))) return` before starting its mutation (SMS import/launch, calendar cancel/email actions, availability/lookup archive, people archive/form guard, dealer archive).
