## Codex Adversarial Review

Codex returned six findings, summarized below. Verbatim text was applied to the in-cycle fixes; see the action-required block.

### High
1. **Concurrent edits on an already-sent quote misclassified as successful Re-sends.** A `setQuoteInputs` save by user B between user A's pre-load and A's Re-send UPDATE left `status='sent'` but bumped `updatedAt`. Pre-fix path returned `ok:true` to A even though no PDF/email/audit fired. **Fixed in-cycle at `4caa049`** by pre-loading `sentAt` and only collapsing to ok-with-no-side-effects when re-select shows `sentAt` advanced past the preload; otherwise return the retry error. New test asserts the concurrent-edit case rejects.
2. **Re-send failure after the guarded UPDATE leaves `sent_at` advanced without a delivered/audited send.** A GCS overwrite or email-send failure after the row UPDATE committed resets the validity window (via the derived `isExpired` projection) but no PDF was actually delivered. Acceptance could happen against an undelivered fresh PDF. **Parked as 0046 follow-up.** This is the same family as the pre-existing degraded-send retry gap that 0026's "PDF/email-after-UPDATE failure" branch already acknowledged, and aligns with the plan's OQ #6 framing of concurrent-send race as out-of-scope. The next Re-send brings reality back into sync, so manual recovery exists; a dedicated outbox-style retry path is the real fix. Wiki rewritten at `4caa049` to call out that subsequent Re-send recovers (no longer claims the system is stuck).

### Medium
1. **Send history renders the current recipient denorm for every historical send row.** Multi-row Send-history made the existing single-recipient denorm wrong when contacts rotate between sends. **Fixed in-cycle at `4caa049`** by adding `sentToEmail` + `sentToFirstName` to the `quote.sent` audit payload and rendering per-row recipient from the payload with a fallback to the row-level denorm for pre-0046 rows.
2. **TypeScript pgEnum order didn't match the migration / snapshot order.** Schema array carried `msa.*` at the end; live Postgres + snapshot put it before `campaign.cancelled` (per `0019` ALTER TYPE ADD VALUE BEFORE). **Fixed in-cycle at `4caa049`** by reordering the TS array to mirror the live enum and adding a comment explaining the rule.

### Low
1. **Wiki retry description claimed "stuck on the idempotent concurrent-send-won path."** Stale wording. **Fixed in-cycle at `4caa049`**: rewrote to say a subsequent Re-send retries, but the failed attempt advanced row state without a delivered audit row.
2. **Hydration drift hypothesis in the closed Re-send dialog** — `formatValidUntil` uses `Date.now()` in `ConfirmSendDialog` content rendered while `open={false}`. Codex couldn't verify Base UI's closed-content DOM behavior from repo code. **Caveat / parked.** The banner copy was switched to absolute "Sent on YYYY-MM-DD" specifically to avoid this class of issue; the dialog content is hidden by default with Radix's unmount-on-close, so the practical risk is low. If a hydration warning surfaces in dev console, switch to opening-time computation.

### Not an issue (false-positive reduction by Codex)
- `setQuoteInputs` optimistic-lock classification correctly distinguishes terminal status → friendly error, non-terminal-status updatedAt-mismatch → retry.
- First-send MSA gate cost — the MSA SELECT is gated on `if (draft.sentAt != null)`; first-send skips it.
- MSA active/terminated/expired gate semantics — the query filters `status='pending'` and `dropboxSignDocumentId != null`.
- Simultaneous Re-send with same preloaded `updatedAt` — Postgres row-locking serializes; second UPDATE misses on the predicate.
- `quote.edited` input-only saves — dirty fields are derived only from priced output; non-priced changes correctly skip the audit row.
- `hashLineItems` 16-hex prefix collision — not practical here since users submit bounded inputs, not arbitrary jsonb.
- Audit-action enum placement in the migration is correct (`ADD VALUE … AFTER 'quote.sent'`).
