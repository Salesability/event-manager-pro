## In-cycle fix — `d2b6654`

`markThreadRead` now requires a `seenThrough` ISO timestamp (the thread's serialized `lastMessageAtIso` snapshot) and stamps `lastReadAt = seenThrough + 1ms` instead of `now()`; both call sites (inbox read-on-open, console Mark read) pass it. An inbound newer than the rendered snapshot stays unread.

**The +1ms matters:** the first cut of the fix wrote the ISO value verbatim and the browser re-verify caught a real defect — `timestamptz` stores microseconds, the ISO snapshot carries milliseconds, so `last_read_at` truncated *below* the very message the client rendered (`…39.799` < `…39.799802`) and the thread read as unread forever in SQL while the JS row derivation (driver parses to ms) said read — badge and list disagreeing. The round-up restores agreement across all three derivations (verified: DB cleared, chip gone, badge gone, SQL count 0).

Also in `d2b6654`: the flaky integration test now asserts a thread-scoped SQL unread derivation through the same lifecycle instead of global-count deltas — the parallel-suite flake that dogged every full run this session is gone at the root.
